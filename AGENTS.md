# AIShortvideo Project Rules

本项目是 AI 短视频小说系统，目标是先把小说生成质量做好，再逐步接入语音、视频和发布能力。

## 当前工作方式

- 当前阶段优先沉淀需求、架构和实现标准；除非用户明确要求，不要擅自生成原型或展开业务代码实现。
- 需求讨论内容需要及时落到 `docs/`，不要只停留在聊天记录里。
- 需求设计和原型设计要先按完整产品形态展开，尽量把完整流程、页面、状态、异常和运营闭环想清楚；实现阶段再拆 MVP、第一期、第二期。
- 后续研发推进采用“需求主控会话 + 全栈研发会话 + 测试会话”的协作模式，具体规则见 `docs/development-coordination.md`。
- 当前需求设计会话负责需求设计、任务拆解、研发指挥、测试验收、进度把控和复盘；研发和测试会话对需求主控会话负责，用户只需要看主控会话的结论。
- 做新功能前，先确认它属于热点分析、小说系统、生成任务、视频系统、模型配置、权限/租户中的哪个模块。
- 视频系统早期按简单承载层设计，小说系统质量优先级最高。
- 实施分期时，视频系统第一阶段可以优先视频列表和引用状态，不做复杂视频详情、AI 分镜或平台自动发布。
- 小说需求和原型范围不是普通列表 CRUD，而是从小说列表进入后，到视频设计之前的完整创作闭环：设定、大纲、章节目录、试写、审稿、重写、批量正文、全书审稿和待视频化确认。

## 技术栈

- 前端：Vue 3 + Vite + TypeScript + Element Plus 最新稳定版。
- 后端：Node.js + TypeScript + Fastify。
- 数据库：MySQL。
- 数据访问：Prisma。

## 后端实现硬约束

后端实现前必须阅读并对照：

- `docs/architecture.md` 中的 `数据库设计与 SQL 标准`。
- `docs/architecture.md` 中的 `后端研发规范`。
- `docs/backend-implementation-checklist.md`。

不符合这些标准的实现不要继续推进，先调整设计或代码结构。

## 前端实现硬约束

前端实现前必须阅读并对照：

- `docs/architecture.md` 中的 `前端设计原则`。
- `docs/architecture.md` 中的 `前端研发规范`。
- `docs/frontend-implementation-checklist.md`。

前端不能只追求页面能显示，还要保证小白用户能看懂当前状态、下一步动作和失败处理方式。

## 前端分层规则

- `pages` 只负责页面组合、路由参数和主要交互入口。
- `components` 放模块内可复用 UI，不要把接口请求散在组件深处。
- `services` 负责模块接口封装，页面不得直接调用裸 `fetch` 或 `axios`。
- `model` 放 TypeScript 类型、接口 DTO 和视图模型转换。
- `constants` 放状态、枚举、颜色、动作文案和字段配置。
- `composables` 放列表查询、任务轮询、创建向导、列表行推荐动作等页面逻辑。
- 核心创作流程不要过度配置化；列表和后台配置页可以适度配置化。

## 后端分层规则

- `routes` / `controllers` 只做请求校验、权限上下文、参数转换和响应包装。
- `services` 负责业务编排、状态流转、事务边界、任务调度和操作日志。
- `domain` 放状态机、枚举、纯业务规则和值对象。
- `repositories` / `data-access` 负责 Prisma 数据访问，提供明确方法。
- `providers` / `clients` / `integrations` 封装外部能力，例如大模型、TTS、视频工具、对象存储、短视频平台、Redis。
- `workers` / `tasks` 处理异步任务和重试。
- 不要让 service 到处散落原始 Prisma 调用。
- 不要让 route 层承载复杂业务逻辑。

## 数据与审计规则

- 重要创作资产不能静默覆盖，必须通过版本记录保存历史。
- AI 生成结果默认先进入候选结果或新版本，用户确认后才成为当前正式内容。
- 高风险操作必须保存原因、影响范围和操作日志，例如清空后续章节、确认低分章节、归档小说、发布提示词模板。
- 涉及章节重写、视频引用章节修改、批量处理时，需要记录影响快照。
- AI 审稿阈值、内容安全严格程度、原创化严格程度、重写强度、市场导向程度、视频化倾向程度和自动处理程度不能写死在业务代码里，必须通过策略配置或共享枚举读取。
- API Key、密钥、外部平台 token、完整提示词、完整模型响应不能写入普通日志、任务摘要或前端响应。

## 接口与任务规则

- 接口需要统一响应结构、统一错误码和统一分页结构。
- 每个接口必须有输入 schema 校验。
- 用户点击类高风险动作需要幂等、防重复和版本校验。
- AI 生成、审稿、影响评估、TTS、视频渲染、发布都按异步任务处理。
- 同一小说、同一章节、同一任务对象，同一时间不能执行互相冲突的任务。
- worker 不能绕过用户确认直接修改正式创作资产。

## 前端交互规则

- 小说列表、热点报告列表、任务日志、模型配置、提示词模板等后台列表页可以使用配置化列表模式。
- 需求和原型设计优先覆盖完整小说创作体验：小说列表、创建小说、小说详情工作台、章节详情工作台、视频列表，以及从小说列表到待视频化前的小说生成、审稿、调试和任务进度交互。
- 实施分期时可以优先实现小说列表、创建小说、轻量章节详情工作台和视频列表；小说详情工作台、视频详情页可按排期裁剪，但原型设计阶段需要保留完整形态。
- 创建小说向导、小说详情工作台和章节详情工作台需要优先手写交互体验，不要套普通 CRUD。
- 每个关键页面都要展示当前状态和下一步建议。
- AI 生成任务不能只显示按钮 loading，必须展示任务进度、失败原因和重试入口。
- 高风险操作需要二次确认，确认文案要说明影响范围。
- 状态标签、评分、风险和推荐动作需要使用统一配置，不能散落在页面模板中。
- API Key、密钥、外部平台 token、完整提示词、完整模型响应不能出现在前端页面、控制台日志或本地存储中。

## 文档维护

- 新增后端表、接口、任务、状态、枚举时，需要同步更新相关文档。
- 新增前端页面、核心组件、路由、状态枚举和高风险操作时，需要同步更新相关文档。
- 如果实现中发现文档标准不适合实际项目，先提出调整建议，再修改文档和代码。
- 文档中的标准优先服务项目落地，不为了形式感引入不必要复杂度。
