# 研发协作机制

本文档定义 AIShortvideo 后续研发推进时的多会话协作方式。它是项目协作规则，不替代产品需求、架构文档或测试用例。

## 协作角色

### 需求主控会话

当前需求设计会话是主控会话，对用户负责。

职责：

- 继续维护需求设计、原型口径、模块边界和研发前契约。
- 把用户意图拆成研发任务和测试任务。
- 给研发会话下发任务说明、范围、验收标准和禁止事项。
- 给测试会话下发测试范围、重点风险和验收口径。
- 汇总研发和测试结果，判断是否满足用户目标。
- 向用户汇报进度、风险、取舍和下一步建议。

### 全栈研发会话

全栈研发会话负责具体实现，不直接扩大需求范围。

职责：

- 按主控会话给出的任务说明实现前端、后端、数据库、任务、接口或集成能力。
- 开始实现前读取项目规则、架构文档和相关 checklist。
- 发现需求或架构不清楚时，向主控会话反馈，不自行拍板扩大功能。
- 完成后返回变更文件、验证命令、已知风险和未完成事项。

### 测试会话

测试会话负责独立验证研发结果可靠性。

职责：

- 按主控会话给出的验收口径设计测试重点。
- 验证页面交互、接口契约、状态流转、任务失败、版本保护和高风险操作。
- 对研发结果给出通过、阻塞、需修复或可接受风险结论。
- 输出复现步骤、证据、风险等级和建议修复顺序。

## 信息流

用户只需要和需求主控会话沟通。

默认信息流：

1. 用户提出目标或反馈。
2. 主控会话判断需求、拆分任务、确认验收口径。
3. 主控会话指派研发会话执行。
4. 研发会话完成实现并回报。
5. 主控会话指派测试会话验证。
6. 测试会话输出测试结论。
7. 主控会话综合研发和测试结果，对用户汇报。

研发会话和测试会话可以由主控会话通过 Codex 线程工具沟通。它们的结论需要回到主控会话统一判断，不能直接替代主控会话向用户拍板。

## 主控持续跟进机制

需求主控会话不能只创建研发和测试会话后等待用户提醒，必须主动巡检和推进。

当前采用长期目标驱动，不使用定时器或固定频率自动化。主控会话在恢复、收到用户反馈、研发交付、测试结论或质量提醒时，先读取相关会话最近状态，再决定下一步。

巡检规则：

1. 如果研发会话仍在执行，不打断，只记录进度。
2. 如果研发会话已完成当前任务包，而测试尚未验收，主控会话立即指派测试会话验收。
3. 如果测试会话验收通过，主控会话汇总结论，并按研发任务包计划判断是否派发下一任务包。
4. 如果测试会话验收阻塞，主控会话指派研发会话修复，再要求测试复验。
5. 如果任一会话 idle 且等待明确下一步，主控会话负责补发指令，不等待用户提醒。
6. 主控会话不得重复发送同一指令；发送前必须检查最近会话记录。
7. 主控会话不得跳过测试直接进入下一任务包。

## 需求多 Agent 评审门禁

每个新阶段在派发研发前，必须由主控先形成需求草案，再进行至少三个独立视角的评审：

- 产品流程与普通用户视角：确认用户目标、主流程、状态、异常、下一步提示和不做项。
- 技术架构与可落地性视角：确认数据模型、接口、状态机、幂等、审计、迁移和模块边界。
- 测试与验收视角：确认 fixture、正常/异常场景、阻塞标准、回归范围和环境结论边界。

涉及数据安全、权限、外部平台、真实模型、真实数据库或发布能力时，必须增加安全/工程质量视角。

主控负责处理评审分歧并将最终决策写入正式需求。只有同时满足以下条件，阶段需求才可标记为 `confirmed`：

1. 需求正文、范围、不做项和分包顺序已落到 `docs/`。
2. 产品、架构、测试均完成成稿复核，且不存在未解决的阻塞意见。
3. 验收 fixture、通过标准和阻塞标准可执行。
4. 真实环境未验证的部分有明确边界，不以 mock、静态检查或内存仓储结果代替。
5. 主控获得用户授权后，才向研发会话派发实现。

## 任务下发格式

主控会话给研发会话下发任务时，至少包含：

- 背景和目标。
- 涉及模块。
- 需要阅读的文档。
- 明确要改的范围。
- 明确不要改的范围。
- 验收标准。
- 推荐验证命令。
- 风险点和回报格式。

主控会话给测试会话下发任务时，至少包含：

- 本次要验证的变更范围。
- 业务闭环或核心场景。
- 必测状态和异常。
- 需要重点防止的回归。
- 通过标准和阻塞标准。

## 决策规则

- 用户目标和项目文档优先于研发会话的个人判断。
- 研发会话不能擅自重写需求、改变模块边界或扩大实现范围。
- 测试会话发现阻塞问题时，由主控会话判断是否返工、降级、延后或接受风险。
- 如果研发实现发现文档规则不适合落地，先反馈给主控会话，由主控会话决定是否调整需求文档和架构文档。
- 任何会影响小说核心创作资产、AI 结果确认、状态流转、视频引用快照或安全日志的改动，都必须经过主控会话确认。

## 当前阶段口径

当前项目仍处于需求设计、原型确认和研发前设计阶段。

正式研发开始前，主控会话应先沉淀并持续维护：

- 系统模块边界与衔接节点：`docs/modules/system-module-boundaries.md`。
- 小说系统状态流转与研发契约：`docs/modules/novel-development-contract.md`。
- 核心数据模型：`docs/modules/novel-core-data-contract.md`。
- 接口契约初稿：`docs/modules/novel-api-contract.md`。
- 首批研发任务包：`docs/modules/novel-first-iteration-development-plan.md`。
- 前端页面与路由范围：`docs/prototypes/README.md` 和 `docs/prototypes/novel-system-page-flow.md`。
- 测试验收矩阵：`docs/modules/novel-acceptance-test-matrix.md`。

正式研发开始后，再创建或指派全栈研发会话和测试会话推进对应任务。
