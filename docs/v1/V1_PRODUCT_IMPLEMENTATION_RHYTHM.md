# V1 产品实现节奏

日期：2026-06-07

## 1. 核心原则

项目可以保持大而全的长期蓝图，但实现节奏必须小而准。

每一阶段都只回答一个问题：

> 这一步是否让外行更容易产出内容，或让我们更接近 100 元真实收入？

如果答案不是，就后置。

## 2. 阶段 0：当前已完成

目标：证明系统主链路可运行。

已完成：

- 一句话想法生成故事短视频内容包。
- 内容包包含钩子、脚本、分镜、字幕、标题、封面文案、发布文案。
- 发布前质检。
- Markdown 导出。
- 手工发布记录。
- 案例库。
- 账号画像。
- 模型网关骨架。
- 模型连接测试。
- 3 条模型质量准入测试。
- 收入验证页。
- 20 个触点清单。
- 客户回复、异议、下次跟进。
- 首页项目推进看板。

阶段判断：

- 产品流程已经能跑。
- 真实模型质量未验证。
- 商业收入未验证。

## 3. 阶段 1：真实模型准入

目标：确认系统可以生成可演示内容，而不是只跑 Mock。

必须做：

1. 配置真实模型。
2. 跑设置页的 3 条生成质量准入测试。
3. 如果不通过，先换模型或改提示词。
4. 不继续扩功能，直到至少一个模型通过准入。

验收：

- 3 条准入测试全部通过。
- 每条都是真实模型生成，不是本地 Mock 回退。
- 输出结构完整，评分不低于 38/50。

禁止：

- 不要先做自动剪辑。
- 不要先做自动发布。
- 不要先做 SaaS 登录注册。
- 不要为了接入某个模型大改架构。

## 4. 阶段 2：首个可展示案例

目标：拿到一个可以给潜在客户看的演示案例。

必须做：

1. 用已通过准入的模型生成 3 条候选内容。
2. 人工挑 1 条最像真实可发布内容的。
3. 在编辑页做必要人工修正。
4. 跑发布前质检。
5. 在案例库设为对外演示案例。
6. 收入页确认演示材料摘要引用的是该案例。

验收：

- 有 1 条人工确认过的演示案例。
- 案例标题、钩子、脚本、分镜、字幕、发布文案完整。
- 这个案例可以在 3 分钟内向潜在客户讲清楚。

禁止：

- 不要拿随手测试内容做演示。
- 不要用未通过准入的模型内容做收费交付。

## 5. 阶段 3：首批真实沟通

目标：验证是否有人愿意看、愿意试、愿意付费。

必须做：

1. 在收入页生成今日 5 个联系对象。
2. 用系统话术联系 5 个真实对象。
3. 每个对象记录客户回复、异议、下次跟进。
4. 至少发出 1 个 29 元或 99 元报价。
5. 每天结束时复盘强意向和异议。

验收：

- 5 个真实联系记录。
- 至少 1 个愿意看演示或提供想法。
- 至少 1 个明确报价动作。

禁止：

- 不要把“继续开发功能”当作商业验证。
- 不要只记录测试线索。

## 6. 阶段 4：首个 100 元信号

目标：拿到真实收入或等价试点。

可接受结果：

- 收到 29 元生成服务付款。
- 收到 99 元诊断付款。
- 收到 100 元以上试点定金。
- 对方明确承诺付费试点，并给出下一步时间。

必须做：

1. 把已付款或强意向记录到收入页。
2. 保存客户真实需求。
3. 输出客户想要的内容包。
4. 记录对方是否觉得系统节省时间、降低创作门槛。

验收：

- 收入页显示 `¥100/100`，或至少 1 个等价强付费试点。
- 有客户回复和下一步记录。

## 7. 阶段 5：再决定是否扩功能

只有完成阶段 4 后，才评估是否进入以下方向：

- 竞品样本库。
- 模板库。
- 小说转短视频增强。
- TTS 配音。
- 剪映/剪辑工具导出。
- 自动抓发布数据。
- 部署版或 SaaS。

扩功能判断：

- 如果客户主要抱怨内容质量，优先改模型、提示词、质检和样本库。
- 如果客户主要抱怨不会使用，优先改首页看板、向导和一键导出。
- 如果客户愿意付费但想批量用，才考虑模板库、批量生成和部署。
- 如果没有客户愿意继续，不要急着做 SaaS。

## 8. 每次复盘固定问题

每次会话或阶段复盘都问：

1. 我们现在做的功能是否直接提高外行可用性？
2. 是否直接帮助我们拿到 100 元验证？
3. 是否依赖真实模型质量？
4. 是否有真实客户反馈支撑？
5. 如果今天只能做一件事，最能推进收入验证的是哪件？
