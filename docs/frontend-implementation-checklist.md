# 前端实现检查清单

这份清单用于保证后续前端代码能持续应用当前已经确定的前端研发规范。每次实现新页面、新组件、新接口联调或关键交互前，都需要对照检查。

## 1. 开发前确认

- 是否已经阅读 `docs/architecture.md` 的 `前端设计原则` 和 `前端研发规范`。
- 如果是初始化或重建前端项目，是否使用 `npm create vite@latest apps/admin-web -- --template vue-ts` 生成基础项目，而不是手工拼装 Vue + Vite + TypeScript 工程。
- 是否确认 UI 库使用 Element Plus 最新稳定版，没有继续引入 Ant Design Vue 或旧版 Element UI。
- 当前页面属于哪类：管理列表页、创建向导页、详情工作台页、配置表单页、结果审阅页。
- 当前页面服务哪个模块：热点分析、小说、章节、生成任务、视频、模型配置、提示词模板、权限/租户。
- 当前页面面向小白用户还是高级/管理员用户。
- 是否涉及高风险操作、AI 生成任务、长耗时任务、版本切换或视频引用影响。
- 是否涉及策略与程度配置，例如审稿严格程度、内容安全严格程度、原创化严格程度、重写强度、市场导向程度、视频化倾向程度或自动处理程度。
- 当前工作属于需求/原型设计还是研发实现：需求和原型先看完整产品形态；研发实现再判断第一期、第二期和后续版本。
- 如果进入研发实现，是否属于首期范围：小说列表、创建小说、轻量章节详情工作台、视频列表，以及从小说列表到待视频化前的小说生成、审稿、重写、任务进度和确认交互优先；小说详情工作台、视频详情页可按排期裁剪。

## 2. 页面类型检查

- 管理列表页是否优先使用 `searchSchema`、`tableColumns`、`rowActions`、`batchActions`。
- 创建向导页是否按明确 step 组织，而不是普通表单堆叠。
- 详情工作台是否有顶部概览、流程进度、当前推荐动作和分区内容。
- 配置表单页是否区分普通配置和高风险配置。
- 策略与程度配置页是否支持草稿、测试、发布、回滚和影响范围提示。
- 审阅页是否突出评分、问题、建议动作和确认入口。

## 3. 路由与菜单检查

- 路由是否配置 `title`、`module`、`menuKey`、`keepAlive`、`permissionCodes`、`breadcrumb`。
- 页面标题、菜单选中、面包屑是否由路由 meta 驱动。
- 权限可以后置研发，但路由和按钮是否预留权限字段。
- 是否避免过早引入复杂多标签页、iframe 或微前端通信。
- 从列表进入详情后，是否能保留必要的返回路径和上下文。

## 4. 接口与 Service 检查

- 页面是否通过模块 `services` 调用接口。
- 是否避免在页面或组件中直接写裸 `fetch`、`axios`。
- service 是否有 TypeScript 入参和返回类型。
- 是否把后端 DTO 转成页面需要的 view model。
- 是否使用统一请求层处理错误、登录失效、超时、请求 ID 和提示。
- 长耗时 AI 操作是否只提交任务并返回任务 ID，而不是让页面一直等待。

## 5. 状态与枚举检查

- 小说状态、章节状态、任务状态、审稿评级、影响等级是否使用统一枚举配置。
- 状态标签是否有统一颜色、文案和说明。
- 审稿阈值、风险标签、推荐动作、是否阻塞下一步是否来自后端策略结果或共享配置，避免页面写死分数规则。
- 页面上是否能让小白理解状态，而不只是展示机器值。
- 新增枚举是否同步后端、前端和文档。
- 推荐动作是否和状态一致，例如 `待处理` 章节要有明确处理入口。

## 6. 列表页检查

- 是否有加载状态、空状态、失败状态。
- 是否有分页、筛选、重置、刷新。
- 表格字段是否服务决策，不展示无用字段。
- 是否区分主操作和更多操作。
- 小说列表是否每行只突出一个当前推荐动作。
- 小说列表是否提供展开行、抽屉或弹窗查看摘要，而不是为了摘要提前建设复杂详情页。
- 小说列表是否能承接从设定、大纲、章节目录、试写、审稿、重写、批量正文到全书审稿的推进动作。
- 视频列表是否展示引用小说、引用章节范围、音频/字幕/渲染/发布状态和引用异常。
- 如果是研发首期，视频列表是否先做好待视频化和引用状态承接，并避免把复杂视频设计、AI 分镜和发布流程提前塞进实现范围。
- 视频列表是否每行只突出一个当前推荐动作，例如生成视频、查看进度、重试失败步骤、下载视频、标记发布。
- 列表行是否展示最近任务状态、失败原因和重试入口。
- 高风险行操作是否二次确认。
- 列表查询是否和后端分页、排序、筛选参数一致。
- 是否避免在列表页加载章节正文、完整审稿报告、大 JSON。

## 7. 创建向导检查

- 是否显示当前步骤、步骤目标、下一步按钮。
- 是否有“帮我推荐”或默认推荐，避免小白卡住。
- 是否允许小白不填专业内容，直接使用系统推荐生成方向。
- 高级设置是否默认折叠。
- 方向生成、融合、确认是否能展示评分、风险和推荐理由。
- 方向生成是否按异步任务展示进度、失败原因和重试入口。
- 确认创建后是否回到小说列表，并高亮该小说的下一步推荐动作。
- 失败时是否提供重试、调整条件、换推荐方案等入口。
- 确认创建前是否让用户知道会生成什么项目。

## 8. 详情工作台检查

小说详情工作台和章节详情工作台都属于完整产品形态，需要在需求和原型中设计清楚；研发首期可以先把章节详情做轻量版，小说详情按排期裁剪。

- 顶部是否展示核心概览：状态、进度、质量分、市场分、视频引用状态、推荐动作。
- 是否展示完整创作流程 step。
- 当前阻塞点是否明显，例如章节待处理、审稿未通过、任务失败。
- 章节列表是否能快速看出哪些章节正常、哪些章节待处理。
- 点击章节是否进入能处理问题的详情工作台，而不是只查看文本。
- 是否能查看任务、审稿、版本和影响记录。

## 9. AI 任务交互检查

- 是否有任务进度，而不是只显示按钮 loading。
- 是否展示当前任务阶段、失败原因、可重试动作。
- 是否避免同一对象重复触发冲突任务。
- 任务完成后是否明确告诉用户下一步。
- 任务失败后是否允许重试、换模型、调整参数或取消。
- 任务日志详情是否隐藏敏感信息。

## 10. 高风险操作检查

- 是否有二次确认。
- 确认文案是否说明影响范围。
- 是否要求填写原因，例如归档小说、清空后续章节、确认低分章节。
- 视频已引用章节被修改时，是否提示影响视频。
- 已发布视频、引用异常视频、已有导出文件的视频删除或重新生成时，是否二次确认。
- 是否避免普通编辑按钮直接覆盖正式内容。
- 是否有成功、失败和取消反馈。

## 11. 组件抽象检查

- 新组件是否真的跨页面复用，还是只属于当前页面。
- 通用组件是否保持职责单一，例如 `SearchPanel` 只管搜索，`DataTable` 只管表格和分页。
- 创建小说向导、章节工作台、审稿调试页是否避免被过度配置化。
- 组件 props 是否有明确类型。
- 组件是否避免直接依赖具体业务接口。

## 12. 用户体验检查

- 页面是否直接进入可操作状态，不做无意义欢迎页。
- 小白用户是否能看懂当前要做什么。
- 错误提示是否包含下一步建议。
- 按钮文案是否具体，例如“生成前三章”优于“提交”。
- 页面信息密度是否适合后台工具，不做营销式大图大卡片。
- 移动端不是首要目标，但窄屏下文字和按钮不能明显重叠。

## 13. 安全与敏感信息检查

- 前端页面、控制台日志、本地存储中是否没有 API Key、密钥、token。
- 是否避免展示完整提示词和完整模型响应。
- 管理员配置页是否不显示 API Key 明文。
- 下载、上传、外部链接是否有明确来源和风险提示。
- 错误响应是否不会把后端堆栈或敏感配置展示给用户。

## 14. 测试与验证检查

- 核心页面是否至少能通过 typecheck 和 build。
- admin-web 构建是否通过 `npm run build:budget -w apps/admin-web`。该门禁会重新执行 build、检查 entry/main chunk 预算、检查主要 route chunk 预算，并只允许已记录的第三方 warning 指纹。
- 当前 admin-web build budget 基线：路由懒加载后 entry chunk 约 1.00 MB / gzip 324 kB，正式预算为 entry <= 1.10 MB、route chunk <= 200 kB。预算超限时应继续拆分页面或移除不必要首包依赖，不能通过调高 Vite `chunkSizeWarningLimit` 掩盖。
- 当前仅临时白名单 `@vueuse/core/dist/index.js` 两处 Rolldown `INVALID_ANNOTATION` 指纹：`3362:1`、`5780:23`。该白名单是第三方依赖治理债，不代表永久忽略；新增 warning 必须复核并更新文档或修复。
- 关键 composable 是否有可测试的纯逻辑。
- 研发首期是否覆盖创建小说、小说列表、轻量章节详情、视频列表、列表行推荐动作和任务进度的基本交互验证计划。
- 研发首期是否覆盖从小说列表推进到待视频化前的核心链路验证：生成设定、生成大纲、生成章节目录、试写、审稿、重写、批量正文、全书审稿。
- 小说详情、视频详情可以按实施排期补完整验证计划，但原型阶段需要考虑完整页面状态和核心交互。
- 轻量章节详情工作台需要覆盖正文查看、审稿问题卡片、重写优化、候选版本确认和返回小说列表。
- 接口 mock 或联调失败时，页面是否有兜底状态。
- 完成后是否能说明这个页面如何符合前端研发规范。

## 15. 不照搬旧项目检查

- 是否避免照搬 Vue 2、Vuex、mixin 和旧版 Element UI 写法。
- 是否避免照搬字符串 apiName 请求方式。
- 是否避免照搬微前端、iframe、多系统 tab 通信。
- 是否避免把所有页面都配置化。
- 是否避免为了抽象而牺牲小说创作核心体验。
