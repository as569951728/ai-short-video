# 后端实现检查清单

这份清单用于保证后续后端代码能持续应用当前已经确定的数据库标准和后端研发规范。每次实现新接口、新任务、新表或高风险操作前，都需要对照检查。

## 1. 开发前确认

- 是否已经阅读 `docs/architecture.md` 的 `数据库设计与 SQL 标准` 和 `后端研发规范`。
- 当前需求属于哪个模块：热点分析、小说、生成任务、视频、模型配置、权限/租户。
- 当前动作是普通查询、普通写入、高风险写入，还是异步任务。
- 是否会影响正式创作资产，例如方向、设定、大纲、章节正文、审稿报告、提示词模板。
- 是否涉及外部能力，例如大模型、TTS、视频工具、对象存储、短视频平台。
- 是否涉及策略与程度配置，例如审稿严格程度、内容安全严格程度、原创化严格程度、重写强度、市场导向程度、视频化倾向程度或自动处理程度。

## 2. API 入口检查

- route 是否只做 schema 校验、权限上下文、参数转换和 service 调用。
- 是否有明确输入 schema。
- 是否使用统一响应结构。
- 列表接口是否使用统一分页结构。
- 是否避免在 route 中写复杂业务逻辑。
- 高风险接口名称是否表达业务动作，而不是通用 `update` 或任意字段 `patch`。

## 3. Service 编排检查

- service 是否负责状态校验、版本校验、事务边界和操作日志。
- 写入前是否检查当前状态是否允许操作。
- 写入正式内容前是否检查当前版本号，避免旧请求覆盖新结果。
- 涉及多表核心写入时，是否明确数据库事务边界。
- 是否避免在数据库事务中执行长时间外部调用。
- 是否把 AI、TTS、视频渲染、发布等长耗时动作转成异步任务。

## 4. Repository 数据访问检查

- 是否通过明确 repository/data-access 方法访问 Prisma。
- 是否避免 service 到处散落原始 Prisma 查询。
- 查询是否明确分页或限制数量。
- 列表查询是否只选择展示需要的字段。
- 是否避免把章节正文、长审稿内容、大 JSON 放进列表查询。
- 是否处理软删除过滤。
- 是否考虑唯一约束、索引和常见查询路径。

## 5. Provider / Client 检查

- 外部能力是否封装在 provider/client/integration 层。
- 是否设置超时、重试和错误转换。
- 是否对返回结果做结构校验和字段裁剪。
- 是否避免把供应商 SDK 细节泄露到 service。
- 日志中是否脱敏 API Key、token、完整提示词、完整模型响应。
- 是否支持后续替换供应商或模型。

## 6. 版本与候选结果检查

- AI 生成结果是否先保存为候选结果或新版本。
- 用户确认后是否才切换为当前正式内容。
- 是否记录版本号、来源任务、模型、提示词模板版本、输入摘要、输出摘要。
- 是否保留旧版本，避免静默覆盖。
- 是否能从当前内容追溯到生成任务和审稿报告。
- AI 审稿、重写影响评估、内容安全检查是否记录当时使用的策略版本。

## 7. 审计与影响检查

- 高风险操作是否写入 `OperationLog`。
- 操作日志是否包含操作者、对象、操作前状态、操作后状态、原因、影响范围。
- 章节重写是否触发影响评估。
- 影响后续章节时，是否按轻微、中等、严重影响规则处理。
- 视频已引用章节被修改时，是否记录视频引用影响。
- 归档、清空、批量处理是否要求用户确认。

## 8. 幂等、防重复与并发检查

- 用户重复点击是否会产生重复数据。
- 是否需要 `idempotencyToken + actionType + requestHash`。
- 是否需要 Redis 短锁或数据库唯一约束兜底。
- 同一小说、章节、任务对象是否会出现冲突任务并发执行。
- 重试任务是否能安全重复执行。

## 9. 枚举与状态检查

- 新状态、新类型、新来源是否定义为统一枚举。
- 枚举是否包含机器值、中文名、说明、是否终态、是否阻塞下一步。
- 审稿阈值、风险程度、自动处理程度是否来自策略配置或共享枚举，而不是写死在 service、worker 或前端页面中。
- 前后端是否共享枚举口径，避免各写一份。
- 状态流转是否清楚，例如章节状态、任务状态、小说创作步骤。
- 枚举新增或废弃是否同步更新文档。

## 10. 任务与 Worker 检查

- 长耗时动作是否创建 `GenerationTask`。
- taskType 是否清晰，输入输出结构是否固定。
- 任务过程是否写入事件日志。
- 任务失败是否保存错误码、错误信息和可重试建议。
- 是否支持取消、重试、换模型重试。
- worker 是否避免绕过用户确认直接修改正式资产。

## 11. 日志与敏感信息检查

- 是否有结构化日志，能定位 novelId、chapterId、taskId、action、duration、result。
- 是否区分运行日志和业务操作日志。
- 是否避免记录 API Key、密钥、外部平台 token。
- 是否避免普通日志保存完整提示词、完整模型响应、外部素材全文。
- 失败日志是否足够定位问题，但不会泄露敏感信息。

## 12. 测试检查

- provider/client 是否覆盖参数组装、错误转换、超时和重试。
- service 是否覆盖状态流转、版本切换、幂等、防重复、审稿阻塞。
- repository 是否覆盖关键查询、分页、过滤、软删除和唯一约束。
- AI 真实调用是否独立为 smoke，不作为普通自动化测试默认依赖。
- 核心闭环是否至少有接口级验证：创建小说、生成方向、确认方向、生成章节、章节审稿、章节重写影响处理。

## 13. 完成前自检

- 是否违反 `AGENTS.md` 的项目规则。
- 是否需要同步更新产品需求、架构、模块文档或接口说明。
- 是否有未处理的高风险操作审计。
- 是否有新增枚举但未同步前端或共享类型。
- 是否有新增数据库字段但没有说明业务含义。
- 是否可以向用户说明“这个实现如何符合数据库标准和后端研发规范”。
