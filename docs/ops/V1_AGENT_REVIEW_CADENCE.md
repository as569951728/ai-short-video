# V1 阶段性 Agent 评审机制

日期：2026-06-07

## 1. 目的

Agent 评审不是为了让方案更复杂，而是为了防止项目跑偏。

每次评审只围绕两个项目铁律：

1. 简单易用：外行用户能 30 分钟内产出可发布内容。
2. 能赚到钱：V1 可用后 14 天内拿到不少于 100 元真实收入或明确付费试点。

## 2. 何时拉 Agent

### 2.1 阶段闸门前

以下节点必须拉 agent：

- 真实模型准入前后。
- 首个演示案例生成后。
- 第 5 个真实联系对象完成后。
- 第 10 个访谈记录完成后。
- 第一个付费或强意向出现后。

### 2.2 出现跑偏信号时

以下情况立即拉 agent：

- 连续 2 次会话都在扩功能，没有推进真实模型或客户沟通。
- 想做自动发布、自动剪辑、SaaS、模板市场等后置功能。
- 模型准入没过，但开始讨论商业化包装。
- 有 3 个以上潜在客户无反馈，需要判断是话术、案例、客户选择还是交付问题。

## 3. 固定 Agent 角色

### 3.1 产品节奏 Agent

职责：

- 判断当前研发是否符合 V1 节奏。
- 判断功能是否对外行简单易用。
- 判断哪些功能应该后置。

输出：

- 是否跑偏。
- 下一阶段最重要的 3 个产品动作。
- 应继续后置的功能。
- 阶段验收标准。

### 3.2 商业验证 Agent

职责：

- 判断当前系统是否足够支撑 100 元验证。
- 判断销售话术、报价、触点和线索记录是否可执行。
- 判断下一批运营动作。

输出：

- 最大商业堵点。
- 当前销售包是否足够落地。
- 接下来 7 天执行动作。
- 不跑偏指标。

### 3.3 模型与技术适配 Agent

职责：

- 判断模型网关和准入测试是否足够。
- 判断模型选择是否符合 V1 目标。
- 判断是否需要新增 adapter 或保持 OpenAI-compatible。

输出：

- 模型接入优先级。
- 当前技术设计是否支持替换模型。
- 最大技术风险。
- 下一步最小改动建议。

### 3.4 内容质量 Agent

职责：

- 判断生成内容是否像可发布内容。
- 判断钩子、冲突、情绪、口语化、分镜是否达标。
- 判断演示案例是否可以对外展示。

输出：

- 内容是否可发布。
- 最大 AI 痕迹。
- 必须人工修改的点。
- 是否允许设为演示案例。

## 4. 当前已启动的一次评审

2026-06-07 已启动：

- 产品节奏评审 agent。
- 商业验证评审 agent。
- 模型与技术适配评审 agent。

本次评审目标：

- 判断当前 V1 是否继续围绕简单易用和 100 元验证。
- 判断下一步是否应该优先真实模型准入和首个演示案例。
- 判断是否需要调整当前模型选择和技术路线。

产品节奏评审结论：

- 当前 V1 节奏总体没有跑偏。
- 当前已经到达“继续堆功能就会跑偏”的临界点。
- 下一阶段必须从“做系统”切到“用系统拿真实反馈”。
- 下一阶段验收不看新增功能数，只看：
  1. 真实模型 3 条质量准入全部通过。
  2. 生成 3 条真实候选内容，人工选出 1 条并设为演示案例。
  3. 完成 5 个真实潜在客户沟通，并在收入页记录客户回复、异议和下一步。
  4. 至少发出 1 个明确报价：29 元、99 元或 100 元试点定金。

商业验证评审结论：

- 当前研发进度足够开始 100 元验证，但还不足以证明 2 周内一定能赚到 100 元。
- 最大商业堵点不是收入页或销售话术，而是还没有 1 条真实模型生成、人工确认、可对外演示的案例。
- 收入页和销售包已经足够启动，不需要继续等功能完善。
- 接下来 7 天应按以下节奏执行：
  1. 第 1 天：配置真实模型，跑 3 条质量准入测试。
  2. 第 2 天：生成 3 条候选内容，挑 1 条设为演示案例。
  3. 第 3 天：联系 5 个真实对象，至少发出 1 个报价。
  4. 第 4 天：根据客户回复修正案例和话术，不新增大功能。
  5. 第 5 天：再联系 5 个对象，累计 10 个访谈记录。
  6. 第 6 天：对强意向对象交付 1 条低价样稿或诊断，推动收款。
  7. 第 7 天：复盘是否有强意向或付款；如果 0 强意向，优先查客户选择、案例质量和话术，不继续堆功能。

模型与技术适配评审结论：

- 当前技术设计足够支持 V1 的单模型替换验证，但还不够支持真正多模型路由产品化。
- 最大技术风险是模型输出结构不稳定，导致用户以为系统不可用，或拿 Mock / 低质量内容做商业验证。
- 下一步最小技术改动应优先做：
  1. 服务端请求加入 JSON mode / `response_format`。
  2. 增加 `max_tokens`，避免模型 JSON 被截断。
  3. 新增最小 ModelRun 本地记录。
  4. 同步 `V1_MODEL_GATEWAY.md`，避免文档状态漂移。
- 暂不做复杂多模型路由，先接 1 个高质量主模型，再接 1 个低成本备用模型。

方向一致性复盘结论：

- 当前所做事情和大方向相符，V1 已经从方案、Mock 闭环推进到真实模型准入。
- 当前所做事情和阶段目标相符，已经服务于“外行可用”和“100 元收入验证”两个铁律。
- 成功最关键的三个动作：
  1. 用真实模型生成 3 条候选内容，人工挑出 1 条演示案例。
  2. 用演示案例联系 5 个真实对象，并记录真实回复。
  3. 根据真实反馈只修最影响成交的点。
- 失败最可能的三个原因：
  1. 继续做功能，但没有真实客户沟通。
  2. 演示案例质量不够，导致客户第一眼不信。
  3. 目标客户选错，导致验证信号失真。

## 5. Agent 评审后如何使用

评审意见只做决策输入，不自动推翻项目节奏。

处理规则：

1. 如果多个 agent 都指出同一个跑偏风险，优先修正。
2. 如果 agent 建议扩功能，必须证明该功能直接推进 100 元验证。
3. 如果建议互相冲突，优先选择最小闭环、最快验证客户反馈的方案。
4. 每次采纳或拒绝 agent 建议，都更新 `PROJECT_BOARD.md`。

## 6. 固定评审问题

每次拉 agent 都带上这 6 个问题：

1. 当前做法是否直接帮助外行使用？
2. 当前做法是否直接帮助 14 天赚到 100 元？
3. 当前最大跑偏风险是什么？
4. 下一步最小但最有效的动作是什么？
5. 哪些功能必须继续后置？
6. 本阶段通过或失败的验收标准是什么？
