# 小说系统首批研发任务包 v1

## 文档目的

本文档把小说系统首期研发拆成可交给全栈研发会话执行、可交给测试会话验收的任务包。

研发会话执行前必须阅读：

- `AGENTS.md`
- `docs/modules/novel-development-contract.md`
- `docs/modules/novel-core-data-contract.md`
- `docs/modules/novel-api-contract.md`
- `docs/backend-implementation-checklist.md`
- `docs/frontend-implementation-checklist.md`

测试会话验收前必须阅读：

- `docs/modules/novel-acceptance-test-matrix.md`
- 本文档每个任务包的验收口径。

任务包 5 验收和任务包 6 派发前，还需要阅读：

- `docs/modules/novel-task-packages-1-6-review.md`

## 研发节奏

首批研发不追求一次性做完整产品，而是按以下顺序推进：

1. 基础工程和契约地基。
2. 小说列表和创建草稿。
3. 方向、设定、大纲、章节目录候选与采用。
4. 任务系统和 mock AI provider。
5. 试写与章节详情工作台。
6. 批量正文、章节重写和影响评估。
7. 全书审稿、完成确认和待视频化。
7.5. 真实模型接入小包 M1：DeepSeek Provider 首期接入。
8. 视频引用快照和引用异常。
9. 视频生成工作台、预览、调试和导出。
10. 人工发布记录和数据回填。
11. 短视频单元和系列。
12. 运营复盘和高级视频能力。

每个任务包完成后都要交给测试会话验收，再进入下一个任务包。需求主控会话负责确认范围变更和优先级调整。

说明：小说环节产品验收通过后，用户确认先接真实模型。M1 是插入在任务包 7 和任务包 8 之间的小包，不占用任务包 8 的视频引用范围。M1 详细设计见 `docs/modules/model-integration-m1-deepseek-provider.md`。

## 任务包 0：工程地基与共享契约

目标：让前后端能稳定运行，并建立共享枚举、统一响应、接口 schema 和测试基础。

后端交付：

- Fastify 应用启动。
- 统一响应结构和错误结构。
- 统一错误码。
- 请求 ID 和基础日志。
- Prisma schema 初稿和数据库连接。
- 共享枚举：小说生命周期、创作阶段、阶段状态、任务状态、版本状态、过期等级、风险等级、影响等级。
- 基础 seed：默认租户、默认用户、默认策略版本、默认提示词模板、mock provider 配置。

前端交付：

- Vue 3 + Vite + TypeScript + Element Plus 可运行。
- 路由和基础管理布局。
- API client、错误处理、分页结构。
- shared constants/types。
- mock 数据切换或后端接口切换方式。

验收：

- 前端可打开 `/novels`。
- 后端健康检查可用。
- 前端能请求后端并展示统一错误。
- 枚举文案前后端一致。

不做：

- 不实现完整 AI 配置后台。
- 不做登录权限页面。

## 任务包 2：方向候选、采用和阶段推进

目标：让小说从草稿进入方向确认，并通过采用方向推进到设定阶段。

后端交付：

- `POST /novels/:novelId/directions/generate`
- `POST /novels/:novelId/directions/fuse`
- `POST /novels/:novelId/directions/:versionId/optimize`
- `POST /novels/:novelId/directions/:versionId/adopt`
- CreativeVersion 方向版本。
- AssetDecisionRecord。
- GenerationTask 方向任务。
- mock AI provider 生成 3-5 个方向候选。

前端交付：

- 创建向导或详情工作台中的方向候选展示。
- 方向评分、推荐理由、风险标签。
- 采用、融合、优化入口。
- 低分采用确认和原因输入。

验收：

- 重复点击生成只产生一个活跃方向任务。
- 方向结果先是候选，不自动成为当前方向。
- 采用方向后旧候选状态正确，小说进入 `setting`。
- 低分方向采用必须记录原因。

不做：

- 不接真实热点爬虫。
- 不要求方向质量达到最终生产水平，先跑通契约。

## 任务包 5：试写闭环和章节详情工作台

目标：跑通前 1-3 章试写、审稿、问题处理和进入批量正文门禁。

后端交付：

- `POST /novels/:novelId/trial/generate`
- `POST /novels/:novelId/trial/confirm`
- `GET /novels/:novelId/chapters/:chapterId`
- 默认生成 3 个完整第 1 章候选，高级策略可调整为 2、3 或 5 个。
- 用户选中的第 1 章候选成为第 2-3 章试写的基础版本，但不直接锁定为全书正式正文。
- 第 2 章硬失败时暂停生成第 3 章；普通低分继续生成第 3 章，并由试写总评统一判断。
- 记录评分策略版本、评分证据、用户选择反馈和试写结果回流。
- 试写通过或用户确认风险继续后，生成 `BodyGenerationStrategySnapshot`，作为后续批量正文生成输入。
- ChapterContentVersion。
- ChapterFeatureCard。
- TrialRun 和 TrialChapterResult。
- ReviewReport 章节和试写总评。

前端交付：

- 第 1 章候选对比区，展示 3 个完整候选、横向评分、推荐理由和“选这个继续试写”入口。
- 候选评分证据和风险确认入口。
- 章节详情工作台。
- 章节正文、章节摘要/特性卡片、审稿问题和候选版本展示。
- 试写总评。
- 正文生成策略快照预览，展示后续批量正文会继承的写法、节奏、风险和加强复审规则。
- 试写通过、继续优化、强制通过确认。

验收：

- 章节目录未确认不能试写。
- 默认试写先生成 3 个完整第 1 章候选，候选选择前不能继续生成第 2-3 章。
- 选中候选只作为试写基础版本，试写总评通过或用户确认风险继续前不能静默成为批量正文基础。
- 第 2 章综合分、承接第 1 章、主线连续性或人物一致性出现硬失败时暂停，不能继续生成第 3 章。
- 评分结果必须带 `scoringStrategyVersion`、维度分、证据说明、扣分点和门禁结论。
- 用户选择、未采纳 AI 推荐、风险继续原因和试写总评结果必须可追溯，用于后续质量调试。
- 试写通过或风险继续后必须生成版本化正文生成策略快照；修改试写章节或上游结构后旧快照必须过期或重新生成。
- 试写正文和审稿结果可追溯到任务和版本。
- 低分试写不能静默进入批量正文。
- 强制通过必须记录原因。

不做：

- 不做复杂富文本编辑器。
- 不做视频首条测试生产，只保留建议数据。

## 任务包 6：批量正文、章节重写和影响评估

目标：跑通正文批量生成、单章重写、候选采用和后续影响处理。

详细设计见 `docs/modules/novel-task-package-6-detailed-design.md`。任务包 6 的核心口径是：分批生成、每章审稿、失败暂停、修复进章节详情、采用后做影响评估。

后端交付：

- `POST /novels/:novelId/chapters/batch-generate`
- `POST /novels/:novelId/chapters/:chapterId/generate`
- `POST /novels/:novelId/chapters/:chapterId/rewrite`
- `POST /novels/:novelId/chapters/:chapterId/content-versions/:versionId/adopt`
- `POST /novels/:novelId/chapters/:chapterId/impact-assessments`
- `POST /novels/:novelId/impact-cases/:impactCaseId/resolve`
- ImpactCase。
- LongTermMemory。
- 批量父子任务。
- 默认每批生成 5 章。
- 风险继续进入批量正文时，前 10 章开启加强复审。
- 每章生成后生成章节摘要/特性卡和单章审稿。
- 每批完成后生成批次总结，作为下一批输入。

前端交付：

- 批量正文进度。
- 章节状态列表。
- 批次总结和下一批注意事项。
- 章节候选版本采用。
- 影响评估摘要和处理入口。
- 高风险影响确认。

验收：

- 批量正文必须读取已确认的最新 `BodyGenerationStrategySnapshot`。
- 默认 5 章一批，不能一键无条件生成全书。
- 风险继续时前 10 章必须逐章严格审稿。
- 每批完成后必须有批次总结。
- 单章硬失败时暂停当前批次，不能继续生成后续依赖章节。
- 批量任务部分失败时已完成章节保留。
- 失败章节阻塞全书审稿。
- 章节重写候选不直接覆盖当前正文。
- 采用章节重写后触发影响评估和审稿过期。
- 中等或严重影响未关闭时不能进入全书审稿。

不做：

- 不做自动批量修复所有后续章节。
- 不做复杂版本可视化 diff，首期可用摘要对比。

## 任务包 7：全书审稿、完成确认和待视频化

目标：跑通小说主链路完成门禁，并生成可被视频系统引用的快照。

- 任务包 6 的终点只能是“正文批量生成阶段完成，待全书审稿”，不能提前创建全书审稿、完成确认或待视频化。

后端交付：

- `POST /novels/:novelId/full-review`
- `GET /novels/:novelId/full-review/latest`
- `POST /novels/:novelId/full-review/:reviewId/resolve-issue`
- `POST /novels/:novelId/full-review/:reviewId/force-pass`
- `POST /novels/:novelId/completion/confirm`
- `GET /novels/:novelId/video-readiness`
- `POST /novels/:novelId/video-readiness/recheck`
- `POST /novels/:novelId/video-readiness/confirm`
- `ReviewReport(reviewLevel=full_novel)`
- FullReviewGate。
- CompletionDecisionRecord。
- VideoReadinessSnapshot。

前端交付：

- 全书审稿结果展示。
- 全书问题卡展示和处理入口。
- 完成确认弹窗。
- 待视频化检查清单。
- 首条视频建议展示。
- 待视频化确认入口。

验收：

- 全书审稿过期不能确认完成。
- 完成确认和待视频化确认是两个动作。
- 待视频化检查失败时小说可保持已确认完成但暂不可视频化。
- 生成待视频化快照后小说进入 `video_ready`。
- `video_ready` 只表示小说可被视频模块引用，不代表已经创建视频项目。

不做：

- 不直接创建视频项目。
- 不做 TTS、字幕、渲染。
- 不做自动发布。

## 任务包 8：视频引用快照和引用异常

目标：视频系统作为下游承接，能基于小说快照创建视频引用，并在小说变化后识别异常。

复盘后口径：任务包 8 只做“视频引用承接层”，不做视频生成。但任务包 8 必须按完整视频系统对象建地基，不能做成一次性的简单列表 CRUD。视频生成、预览调试、发布记录、数据回填、短视频单元和运营复盘分别拆到任务包 9-12。

详细设计见：`docs/modules/video-task-package-8-detailed-design.md`。任务包 8 派发、研发和验收时，以该文档的模块拆分、接口契约、幂等规则、阻塞标准和验收口径为准。

当前收口状态：2026-06-23，P8 前端步骤式承接工作台已通过测试会话复验并收口，复盘见 `docs/reviews/video-p8-acceptance-closure-2026-06-23.md`。当前通过范围是前端产品层和 mock/view-model 闭环；后端持久化、真实接口、幂等和操作日志建议拆为 P8b，不应被误认为 P9/P10 已开始。

后端交付：

- `POST /videos`
- `GET /videos`
- `GET /videos/:videoId/reference`
- `POST /videos/:videoId/reference/recheck`
- VideoProject。
- VideoReference。
- 默认 VideoUnit。
- VideoReferenceIssue。

前端交付：

- 视频列表。
- 引用小说、章节范围和引用状态。
- 引用异常标签和详情。
- 从小说详情跳转视频列表创建视频项目入口。
- 视频列表样式和交互参考小说列表：状态筛选、筛选卡片、表格卡片、行展开、详情/任务抽屉。

验收：

- 只能引用 `video_ready` 小说。
- 视频项目保存章节版本快照。
- 创建视频项目后生成默认视频单元，但不进入生成。
- 小说后续修改不会自动修改视频项目。
- 被引用章节变化后生成引用异常。
- 已发布视频记录不被覆盖。

不做：

- 不做 TTS、字幕、渲染和发布自动化。
- 不做视频生成工作台和完整详情生产能力；任务包 8 只保留引用详情或详情抽屉。
- 不做旁白稿生成、标题封面候选、人工发布记录、24/48 小时数据回填和运营复盘。

## 任务包 8b：视频承接后端持久化和接口接入

详细设计见：`docs/modules/video-task-package-8b-detailed-design.md`。研发执行计划见：`docs/superpowers/plans/2026-06-23-video-p8b-backend-persistence.md`。

## 任务包 9：简单视频生成闭环

目标：在视频项目和引用快照稳定的前提下，跑通“旁白稿 -> TTS 音频 -> 字幕 -> 视觉方案 -> 渲染 -> 系统内预览 -> 不满意可编辑/重生成 -> 确认导出”的首版生成链路。

重要口径：任务包 9 可以生成内容简单的视频，但系统能力不能简单到只能一键生成文件。必须具备预览、编辑、重新生成、版本化和不满意回退路径。

详细设计见：`docs/modules/video-task-package-9-detailed-design.md`。任务包 9 派发、研发和验收时，以该文档的工作台结构、产物版本、依赖过期、任务幂等、不满意闭环、阻塞标准和验收口径为准。

核心交付：

- 视频详情工作台地基。
- 视频生成步骤条。
- 旁白稿版本。
- 旁白稿候选、手动编辑、确认和重新生成。
- TTS 音频版本。
- 音频试听、音色/语速调整和重新生成。
- 字幕版本。
- 字幕编辑、首屏字幕、时间轴摘要和字幕样式。
- 视觉方案版本：首期可以是循环背景素材和画面比例。
- 循环背景渲染任务。
- 渲染视频文件版本。
- 系统内视频预览播放器。
- 不满意原因记录和返回编辑入口。
- 当前视频确认和导出。
- 视频生成任务进度、失败原因、重试和取消。
- 视频产物依赖和下游过期规则。

关键规则：

- 旁白稿确认后才能生成音频。
- 音频变化后字幕和渲染产物过期。
- 字幕变化后渲染产物过期。
- 背景素材或渲染参数变化后渲染产物过期。
- 渲染前必须检查视频引用没有 blocking 异常。
- 未确认的视频不能导出。
- 用户不满意当前视频时，必须能返回旁白、音频、字幕或视觉方案继续调整。
- 生成产物不能覆盖已发布视频使用的版本。
- 重新生成和重新渲染都创建新版本，不覆盖旧版本。

不做：

- 不做自动发布。
- 不做平台数据同步。
- 不做复杂 AI 分镜，但需要保留视觉方案扩展点。
- 不做短视频集自动拆条。

建议拆分：

- 任务包 9a：视频详情工作台地基、步骤条、产物版本、任务、预览框架和依赖过期。
- 任务包 9b：旁白稿调试，包含候选、编辑、确认、重新生成。
- 任务包 9c：TTS 音频调试，包含试听、换音色/语速、重新生成。
- 任务包 9d：字幕调试，包含编辑、首屏字幕、时间轴和样式。
- 任务包 9e：渲染预览导出，包含播放器预览、不满意原因、确认当前视频、重新渲染和导出。

## 任务包 10：人工发布记录和数据回填

目标：让用户可以把导出视频发布到平台后，回到系统记录发布信息和 24/48 小时基础数据，形成最小运营闭环。

详细合同、状态机、数据模型、API 草案、分包和验收矩阵见 `docs/modules/video-task-package-10-detailed-design.md`。2026-07-11 已完成产品、架构、测试、安全四视角需求评审；当前仅完成需求准备，尚未授权 P10 实现。开工前先按 `docs/modules/novel-creation-source-contract.md` 关闭小说 `creationSource` 合同缺口并刷新工作树归因检查点。

核心交付：

- 标记发布。
- 人工发布记录：平台、账号、链接、发布时间、标题、钩子版本、首屏字幕版本、使用视频文件版本。
- 24/48 小时数据回填：播放、完播、点赞、收藏、评论、关注。
- 数据待回填、逾期、样本不足、已记录下一步决策等状态。
- 下一步决策：继续投放、优化标题、优化旁白、换章节、暂停项目。

关键规则：

- 发布记录必须冻结当时的视频文件、音频、字幕、发布标题、前三秒钩子、首屏字幕、结尾悬念和引用版本；封面文案如后续任务包 11 引入，则冻结对应版本。
- 已发布视频引用异常时，不能自动覆盖平台内容。
- 数据不足时复盘结论必须标记低置信度。

不做：

- 不做自动上传。
- 不做平台 API 自动同步。
- 不做高级复盘看板。

## 任务包 11：视频详情和短视频单元

目标：补强视频项目的持续创作体验，让用户能按视频单元管理标题、封面、钩子、分镜和短视频集基础结构。

核心交付：

- 视频详情页完整首版。
- 短视频单元拆分。
- 单元范围调整。
- 标题、封面文案、前三秒钩子候选。
- 分镜脚本或视觉脚本草案。
- 短视频集基础字段。
- 单元级引用快照和产物状态。

关键规则：

- 调整单元范围后，旁白、音频、字幕和渲染产物需要过期或重新生成。
- 标题/封面/钩子候选不自动成为发布版本，用户确认后才进入当前版本。
- 分镜脚本不能直接覆盖已确认视觉方案。

## 任务包 12：运营复盘和高级视频能力

目标：把视频表现回流到小说、热点和系统自我成长模块，并为后续自动发布、外部视频工具和 AI 分镜预留闭环。

核心交付：

- 运营复盘看板。
- 标题/钩子/章节表现对比。
- 小说和热点回流分析。
- 视频 Agent 效果复盘。
- 模型、提示词和策略优化建议。

后置能力：

- 外部视频工具。
- AI 分镜。
- 自动发布。
- 平台 API 数据同步。
- 高级数据看板。

## 并行协作建议

在任务包 0 完成后，可以并行：

- 全栈研发会话继续做任务包 1-3。
- 测试会话准备测试数据、接口验收脚本和前端冒烟用例。
- 需求主控会话复核任务包 4-8 是否需要根据研发发现微调。

不建议并行：

- 未完成任务包 0 前，不要拆多人同时改状态、任务和版本基础契约。
- 未完成任务包 3 前，不要先做全书审稿和待视频化。
- 未完成任务包 7 前，不要做正式视频引用。

## 首批任务完成定义

任务包 0-8 完成后，首批研发才算可进入小范围自用验证：

- 用户能从列表创建小说并推进到待视频化。

- 所有 AI 产物可追溯到任务和版本。

- 测试会话通过主链路、异常链路、幂等、任务冲突和版本过期验收。
