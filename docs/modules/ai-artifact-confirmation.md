# AI 产物确认规则

本文档补齐 `GAP-P0-006`：明确 AI 生成、审稿、重写、影响评估等任务产物，哪些可以自动保存，哪些必须进入候选版本，哪些必须由用户确认后才能成为正式内容。

核心原则：AI 可以多生成、多建议、多评估，但不能静默覆盖关键创作资产。

## 基本概念

### AI 产物

AI 产物指由模型、规则任务或自动化 worker 生成的结果，包括：

- 热点报告。
- 小说方向。
- 审稿报告。
- 设定档案。
- 全书大纲、阶段大纲、章节目录。
- 章节摘要/特性卡片。
- 章节正文。
- 章节重写候选版本。
- 影响评估报告。
- 长篇记忆。
- 全书审稿报告。
- 音频、字幕、视频文件和发布同步记录。

### 正式创作资产

正式创作资产是后续生成会依赖的当前有效内容，例如：

- 已锁定小说方向。
- 当前设定档案。
- 当前全书大纲、阶段大纲和章节目录。
- 当前章节摘要/特性卡片。
- 当前章节正文。
- 当前长篇记忆。

这些内容不能被 AI 任务直接原地覆盖，必须通过版本切换、确认记录和操作日志完成变更。

### 候选产物

候选产物是 AI 生成但尚未成为正式内容的结果。

候选产物可以被：

- 采用。
- 放弃。
- 继续优化。
- 复审。
- 标记为过期。
- 作为复盘数据保留。

## 处理分类

AI 产物按结果处理方式分为四类：

| 分类 | 含义 | 是否需要用户确认 |
| --- | --- | --- |
| 自动记录 | 只作为报告、日志、成本或复盘信息保存，不改变正式创作资产 | 不需要 |
| 自动成为当前版本 | 在严格前置条件下可直接设为当前版本 | 通常不需要，但要可追溯 |
| 候选待确认 | 先保存为候选版本，用户确认后才成为正式内容 | 需要 |
| 高风险确认 | 采用或执行前必须二次确认并填写或选择原因 | 必须 |

默认规则：

- 只要会改变正式创作资产，优先进入候选待确认。
- 只要会影响后续章节、视频引用、完成门禁或低分风险，进入高风险确认。
- 审稿报告、影响评估、任务日志和成本记录可以自动保存，但它们产生的建议动作不能自动改写正文。
- 格式修复在不改变业务含义时可以自动保存；一旦改变剧情、评分、设定或章节事实，就必须进入候选待确认。

## 产物确认矩阵

| 产物 | 默认处理 | 是否可自动成为当前版本 | 采用后副作用 |
| --- | --- | --- | --- |
| 热点报告 | 自动记录 | 可以作为最新报告，但不自动创建小说 | 可被创建小说引用 |
| 小说方向生成结果 | 候选待确认 | 不可以 | 用户选择后进入锁定方向或创建草稿项目 |
| 方向审稿报告 | 自动记录 | 不适用 | 低分时影响推荐动作，不直接改方向 |
| 方向优化/融合结果 | 候选待确认 | 不可以 | 采用后替换锁定方向版本，重新计算下一步 |
| 小说设定档案 | 候选待确认 | 不可以 | 采用后允许生成大纲 |
| 设定审稿报告 | 自动记录 | 不适用 | 低分时阻塞或提示优化设定 |
| 全书大纲 | 候选待确认 | 不可以 | 采用后允许生成阶段大纲 |
| 阶段大纲 | 候选待确认 | 不可以 | 采用后允许生成章节目录 |
| 章节目录 | 候选待确认 | 不可以 | 采用后允许试写或生成章节卡片 |
| 章节摘要/特性卡片 | 自动成为当前版本或候选待确认 | 初次生成可自动；人工编辑或重大变化需确认 | 影响正文生成和重写约束 |
| 章节首次正文 | 自动成为当前版本或候选待确认 | 策略允许时可自动；试写章节建议待审稿确认 | 触发章节审稿和长篇记忆更新 |
| 章节重写/局部优化结果 | 候选待确认 | 不可以 | 采用后触发章节复审、长篇记忆同步和影响评估 |
| 候选版本复审报告 | 自动记录 | 不适用 | 决定是否推荐采用 |
| 章节审稿报告 | 自动记录 | 不适用 | 低分时标记章节待处理或建议优化 |
| 影响评估报告 | 自动记录 | 不适用 | 中等影响可标记受影响章节；严重影响进入用户决策 |
| 后续章节处理方案 | 候选待确认或高风险确认 | 不可以 | 采用后批量创建候选修正任务或清空后续当前引用 |
| 长篇记忆同步 | 自动成为当前版本 | 可以 | 更新后续生成上下文 |
| 全书审稿报告 | 自动记录 | 不适用 | 通过后进入完成候选，未通过时阻塞 |
| 格式修复结果 | 自动记录或自动成为当前结构化结果 | 不改变业务含义时可以 | 修复结构化输出，保留原任务记录 |
| 音频/字幕/视频文件 | 自动记录 | 可以保存文件引用 | 更新视频项目生成状态 |
| 发布记录 | 高风险确认 | 不可以自动发布 | 记录人工发布或平台同步结果 |

## 各类产物规则

### 热点报告

规则：

- 热点报告可以自动保存为报告记录。
- 最新报告可以作为创建小说默认推荐来源。
- 热点报告不能自动创建小说，也不能自动锁定小说方向。
- 用户引用某份热点报告创建小说时，需要记录报告 ID 和版本。

### 小说方向

方向生成通常会输出 3-5 个方向。

规则：

- 每个方向都保存为候选方向。
- 系统可以标记 AI 推荐方向，但不能自动替用户锁定方向。
- 用户选择方向、融合方向或优化方向后，才进入锁定方向流程。
- 低分方向仍允许确认，但必须提示风险并记录确认原因。
- 方向被锁定后，后续设定、大纲和章节都引用锁定方向版本。

### 方向优化和融合

规则：

- 优化和融合结果必须生成新方向版本。
- 新方向不能覆盖原方向。
- 采用前需要展示保留内容、放弃内容、新增内容、评分变化和风险变化。
- 采用后原方向变为历史版本，新方向成为当前锁定方向。
- 如果方向变化较大，后续设定、大纲和章节目录需要重新评估。

### 小说设定档案

规则：

- 设定档案生成后进入候选待确认。
- 用户确认设定后，设定才能成为当前正式设定。
- 设定未确认前不能生成全书大纲。
- 用户修改设定时，必须生成新设定版本。
- 已有大纲或章节时修改设定，需要触发影响评估或提示后续产物可能失效。

### 全书大纲、阶段大纲和章节目录

规则：

- 全书大纲、阶段大纲、章节目录都属于关键结构资产，默认候选待确认。
- 大纲确认前不能生成下游产物。
- 阶段数量调整后，需要重新计算阶段范围和章节目录影响。
- 章节目录确认后才能试写或批量生成正文。
- 修改已确认章节目录时，需要提示会影响章节卡片、正文和已生成视频引用。

### 章节摘要/特性卡片

规则：

- 初次从已确认章节目录生成章节卡片时，可以自动成为当前版本。
- 正文生成后自动补全或更新摘要时，可以自动成为当前版本，但必须保存旧版本。
- 用户手动编辑章节卡片，或 AI 因重写建议大幅修改章节卡片时，需要生成候选卡片。
- 章节卡片变化涉及核心任务、关键事实、伏笔或结尾钩子时，采用前需要提示影响正文和后续章节。

### 章节首次正文

规则：

- 没有当前正文时，首次正文生成可以保存为当前正文版本。
- 试写章节建议在审稿通过或用户确认风险后，才允许进入批量生成。
- 批量正文生成可以按策略自动保存当前正文，但低分章节要标记为 `pending`。
- 首次正文生成后必须自动触发或提示章节审稿。
- 首次正文生成后需要更新长篇记忆。

### 章节重写和局部优化

规则：

- 只要当前章已经存在正式正文，重写和局部优化结果必须进入候选版本。
- 候选版本未采用前，不影响当前正式正文、长篇记忆和后续章节。
- 候选版本需要展示改动摘要、评分变化、问题数量变化、关键事实变化和风险提示。
- 评分没有提升时，默认不推荐采用。
- 采用候选版本后必须触发章节复审、长篇记忆同步和影响评估。

### 审稿报告和问题卡片

规则：

- 审稿报告自动保存。
- 审稿报告不能直接修改正文、大纲或设定。
- 审稿问题可以转成问题卡片和推荐动作。
- 用户点击问题动作后，系统创建新任务并生成候选版本。
- 用户强制通过低分审稿时，必须填写或选择原因。

### 影响评估

规则：

- 影响评估报告自动保存。
- 无影响和轻微影响可以自动同步章节摘要、特性卡片和长篇记忆。
- 中等影响可以自动把受影响章节标记为 `pending`，备注为“受影响未调整”。
- 严重影响不能自动清空后续章节，必须让用户选择处理方案。
- 清空后续章节、逐章处理、返回修改当前候选版本三种选择都要记录原因和影响范围。

### 长篇记忆同步

规则：

- 长篇记忆同步可以自动成为当前版本。
- 同步必须基于当前正式章节正文或已采用候选版本。
- 不能基于未采用候选版本更新当前长篇记忆。
- 如果同步结果改变关键事实、人物关系或伏笔状态，需要触发影响评估。

### 全书审稿

规则：

- 全书审稿报告自动保存。
- 全书审稿通过后，小说进入“待确认完成”或“可候选视频化”，仍需要用户确认完成。
- 全书审稿未通过时，小说不能进入已完成或待视频化。
- 用户强制继续必须填写原因，并保留审稿风险提示。

### 视频相关产物

规则：

- 音频、字幕和渲染文件可以自动保存文件引用。
- 视频项目引用的小说章节发生变化时，必须提示引用异常。
- 平台发布不应默认自动执行；早期可以由用户人工发布后回填发布记录。
- 已发布视频对应小说章节被修改时，系统只提示风险，不自动修改平台内容。

## 用户确认动作

确认类动作统一分为：

| 动作 | 含义 | 常见场景 |
| --- | --- | --- |
| 采用 | 候选版本成为当前正式版本 | 设定、大纲、章节重写 |
| 放弃 | 候选版本保留历史，但不参与当前流程 | 低分候选、不满意方向 |
| 继续优化 | 基于当前候选再创建优化任务 | 候选有提升但仍不够好 |
| 重新生成 | 放弃当前候选，基于当前正式输入重新生成 | 方向、大纲、章节正文 |
| 强制继续 | 明知有低分或风险，仍继续下一步 | 低分方向、低分章节、全书审稿警告 |
| 返回修改上游 | 回到设定、大纲或章节卡片修改输入 | 结构性问题 |

确认操作必须做到：

- 展示要采用的内容摘要。
- 展示当前正式版本和候选版本的差异。
- 展示评分变化。
- 展示风险和影响范围。
- 展示下一步会发生什么。
- 高风险时要求填写或选择原因。

## 必须填写原因的场景

以下场景必须记录原因：

- 低分方向仍确认创建。
- 低分设定、大纲或章节仍强制继续。
- 候选版本评分没有提升但仍采用。
- 候选版本改变关键事实、人物关系、伏笔或结尾钩子。
- 放弃高评分候选版本。
- 强制通过章节审稿或全书审稿。
- 手动确认中等或严重影响无问题。
- 清空后续章节。
- 修改已被视频引用的章节。
- 采用基于旧版本生成的候选产物。

原因可以是枚举加补充说明。枚举建议包括：

- 用户主观偏好。
- 更符合目标受众。
- 更适合短视频表达。
- 成本和时间考虑。
- 认为风险可接受。
- 后续会人工处理。
- 当前候选不符合预期。
- 需要保留原剧情。

## 过期候选规则

候选产物依赖的上游版本发生变化时，需要标记为过期。

常见过期原因：

- 方向变化导致设定候选过期。
- 设定变化导致大纲候选过期。
- 大纲变化导致章节目录候选过期。
- 章节目录变化导致章节卡片和正文候选过期。
- 正式正文变化导致旧候选正文过期。
- 视频引用章节变化导致旧视频生成产物过期。

规则：

- 过期候选默认不推荐采用。
- 用户强制采用过期候选时必须填写原因。
- 强制采用后需要重新触发影响评估或下游同步检查。
- 过期候选不删除，保留用于复盘。

## 状态与任务联动

AI 任务完成后的任务状态和产物确认关系：

| 任务结果 | 任务状态 | 业务状态 |
| --- | --- | --- |
| 自动记录成功 | `completed` | 刷新报告、评分或推荐动作 |
| 自动成为当前版本成功 | `completed` | 刷新当前版本和下一步 |
| 产出候选需确认 | `waiting_confirmation` | 当前阶段进入待确认 |
| 用户采用候选 | `completed` | 候选切为正式，刷新状态 |
| 用户放弃候选 | `cancelled` | 候选保留历史，决策记录为 `discarded`，刷新推荐动作 |
| 高风险确认未完成 | `waiting_confirmation` | 阻塞后续动作 |

本项目统一采用：用户放弃候选时，任务状态记为 `cancelled`，产物决策记录记为 `discarded`，并保留放弃原因。

关键资产采用后需要标记哪些下游候选、审稿报告、门禁和视频化快照过期，完整矩阵见 `docs/modules/novel-asset-adoption-staleness-contract.md`。任务互斥、批量任务和上游版本变化处理见 `docs/modules/novel-task-concurrency-contract.md`。

## 数据记录建议

除了版本表和任务表，建议增加统一的产物决策记录，方便复盘用户采用率和系统自我成长。

### ArtifactDecisionRecord

```text

currentVersionIdAfter

decisionNote

```

`decision` 可取：

- `adopted`
- `discarded`
- `continued_optimization`
- `regenerated`
- `forced_continue`
- `returned_to_upstream`

规则：

- 每次采用、放弃、继续优化、强制继续都要写入决策记录。
- 低风险自动保存结果可以不写决策记录，但必须能从任务和版本追溯。
- 高风险操作同时写入 `OperationLog`。
- 决策记录后续可以转成 `LearningSignal`，用于分析哪些任务、模型、提示词和审稿建议更容易被采用。

## 接口边界

建议通用确认接口只处理任务级结果：

- `POST /tasks/:taskId/confirm-result`：确认采用任务结果。
- `POST /tasks/:taskId/discard-result`：放弃任务结果。
- `POST /tasks/:taskId/continue-optimization`：基于候选继续优化。

关键业务对象也可以保留语义化接口：

- `POST /novels/:novelId/directions/:directionId/select`
- `POST /novels/:novelId/settings/:versionId/adopt`
- `POST /novels/:novelId/outlines/:versionId/adopt`
- `POST /novels/:novelId/chapters/:chapterId/versions/:versionId/adopt`

规则：

- 前端不能通过通用 PATCH 随意切换 `isCurrent`。
- 确认接口必须做版本校验、状态校验、权限校验和幂等控制。
- 高风险确认必须传原因。
- 采用后需要刷新推荐动作、任务状态、业务状态和完成门禁。

## 前端展示要求

确认界面要服务小白用户：

- 用一句话说明“这个结果是干什么的”。
- 用列表展示“采用后会发生什么”。
- 默认展示摘要，不强迫用户读完整长文。
- 提供查看完整内容、版本对比和审稿报告入口。
- 默认只突出一个主推荐动作。
- 高风险操作用明确文案说明影响范围。

常见文案：

- “采用这个设定后，系统会基于它生成全书大纲。”
- “采用这个章节版本后，系统会检查是否影响后续章节。”
- “这次优化分数没有提升，系统不建议采用。”
- “该结果基于旧版本生成，建议重新生成。”
- “继续低分章节会影响小说完成质量，请选择原因。”

## 完成判断

一个 AI 产物确认流程完成，必须满足：

- 产物已保存为版本、报告、文件或日志。
- 如果需要确认，已有明确决策。
- 如果采用候选，当前版本引用已切换。
- 如果放弃候选，候选保留历史且不影响当前版本。
- 如果强制继续，原因已记录。
- 如果有影响范围，已生成影响评估或影响快照。
- 推荐动作和完成门禁已刷新。

## 原型设计要求

后续画原型时，至少需要覆盖：

- 方向选择确认抽屉。
- 设定档案确认抽屉。
- 大纲和章节目录确认抽屉。
- 章节候选版本确认区。
- 低分强制继续确认弹窗。
- 过期候选提示。
- 高风险影响范围确认。
- 采用、放弃、继续优化、重新生成、返回上游修改。

这些确认交互是小说质量控制的核心，不能只用普通“确定/取消”弹窗代替。
