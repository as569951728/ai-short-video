# 章节详情工作台完整形态

本文档补齐 `GAP-P0-004`：用户从小说列表、小说详情工作台或待处理章节入口进入单章后，如何查看正文、理解问题、发起优化、确认候选版本、处理后续章节影响，并最终让章节重新回到可继续推进的状态。

章节详情工作台不是普通文本编辑器，也不是普通 CRUD 详情页。它的核心目标是让小白用户看懂“这一章现在有什么问题、系统建议下一步做什么、操作会不会影响后面章节”。

章节详情中的章节摘要/特性卡片、审稿问题卡和一键优化动作，需要按 `docs/modules/novel-hit-content-integration-matrix.md` 补齐痛点、压迫、反击、情绪回报、切片点、冲突差异和不能丢失的情绪承诺。

## 页面定位

章节详情工作台负责单章创作闭环：

- 查看当前正式章节正文。
- 查看章节摘要和特性卡片。
- 查看 AI 审稿评分、问题和建议动作。
- 基于章节摘要/特性卡片进行一键重写或局部优化。
- 管理候选版本，支持采用、继续优化、放弃。
- 对比正式版本和候选版本。
- 采用新版本后做影响评估。
- 处理中等或严重影响带来的后续章节问题。
- 提示当前章节是否已经被视频引用。

它承接的是“单章问题处理”，不是整本小说结构调整。涉及全书大纲、阶段大纲、章节目录、全书审稿和项目复盘的内容仍回到小说详情工作台。

## 入口场景

| 来源 | 进入原因 | 默认聚焦区域 |
| --- | --- | --- |
| 小说列表推荐动作 | 存在待处理章节、低分章节、重写影响后文 | 问题处理区 |
| 小说详情工作台章节管理区 | 用户点击某章查看或编辑 | 正文和章节摘要 |
| 试写审稿结果 | 前 1-3 章节奏、钩子或爽点不达标 | 审稿问题区 |
| 生成任务结果 | 单章生成失败、重写完成、候选版本待确认 | 任务结果或候选版本区 |
| 影响评估结果 | 当前章重写影响后续章节 | 影响评估区 |
| 视频引用提示 | 已视频化章节发生修改 | 视频引用区 |

进入页面时可以携带 `focus` 参数，例如 `focus=review`、`focus=candidate`、`focus=impact`、`focus=videoReference`。

## 页面布局

完整形态建议采用后台工作台布局，不做沉浸式写作软件。

### 1. 顶部上下文栏

展示：

- 小说名称。
- 章节序号和标题。
- 所属阶段/分卷。
- 章节主状态。
- 状态备注。
- 字数和目标字数。
- 当前评分和评级。
- 视频引用状态。
- 最近任务状态。
- 当前主推荐动作。

要求：

- 顶部始终展示一个最重要的主按钮，例如“生成正文”“AI 审稿”“优化章节”“采用候选版本”“处理影响章节”“确认章节正常”。
- 如果章节存在阻塞问题，顶部需要用通俗文案说明原因，例如“这一章审稿分偏低，建议先优化后再继续批量生成”。
- 如果当前章被视频引用，顶部要展示醒目的引用提示，避免用户误以为只是在改小说。

### 2. 左侧章节导航

展示当前小说的章节列表摘要：

- 章节序号。
- 标题。
- 主状态。
- 评分。
- 状态备注。
- 是否有视频引用。

要求：

- 支持上一章、下一章快速切换。
- 待处理、处理中、低分、有候选版本、影响未处理的章节需要明显标记。
- 左侧列表只加载章节摘要，不加载全部正文。

### 3. 右侧工作区

左侧章节导航之外的区域统一作为右侧工作区，避免“中间正文 + 右上角卡片 + 右侧建议”分散用户注意力。

右侧工作区默认顺序：

1. 章节摘要/特性卡片。
2. 当前正式正文。
3. 建议、优化输入和候选操作。
4. 候选稿、版本历史、影响详情等按需展开。

### 4. 章节摘要/特性卡片

章节摘要/特性卡片放在右侧工作区顶部。

必须包含：

- 正文概览：已有正式正文时，概括当前正文版本、段落数、核心事件和结尾钩子。
- 本章核心任务。
- 主要冲突。
- 爽点。
- 结尾钩子。
- 不能改变的关键事实。

要求：

- 有正文时，用户先看到“当前正文实际写了什么”，再看计划字段。
- 摘要/特性卡是生成新改稿的依据，不是正文下方的补充信息。
- 摘要/特性卡变化要生成版本记录。

### 5. 当前正式正文

展示当前正式版本正文。

内容：

- 正式正文。
- 当前版本号。
- 字数。
- 生成来源。
- 最近采用时间。
- 上一章结尾摘要。
- 下一章目标摘要。

要求：

- 默认展示阅读视图，避免小白误改正式正文。
- 正文区右上角展示字数。
- 正文区需要有“编辑”按钮；点击后切换为编辑状态。
- 保存编辑不能静默覆盖历史版本，必须形成新版本或候选版本。
- 版本标签可以点击查看历史版本。
- 恢复历史版本时，需要以旧版本内容生成新的当前版本，并触发影响评估。
- 长章节可以按段落折叠或定位，但不拆成多个章节。
- 正文区需要能快速跳转到审稿指出的问题位置。

### 6. 建议与优化输入

建议区集中展示当前章最需要处理的信息，标题使用“建议”。

- 主推荐动作。
- AI 审稿摘要。
- 核心问题卡片。
- 候选版本状态。
- 影响评估状态标签。
- 用户补充优化要求输入框。

要求：

- 默认最多展示 3 个核心问题，其余问题折叠。
- 每个问题都要有“问题说明、严重程度、影响、建议动作”。
- 小白文案要避免专业术语，例如用“开头吸引力不够”替代“钩子弱”。
- 用户可以在输入框补充本次优化要求，例如“开头更抓人，不要改旧码头线索”。
- “保存”只保存本次优化要求。
- “生成新改稿”打开候选稿弹窗，不直接覆盖正式正文。
- 任何高风险操作都要二次确认并写原因。

## 核心信息区

### 章节摘要/特性卡片

章节摘要/特性卡片是生成和重写章节的关键依据。

必须包含：

- 一句话摘要。
- 本章核心任务。
- 主要冲突。
- 爽点。
- 情绪关键词。
- 人物变化。
- 关系变化。
- 关键场景或关键信息。
- 伏笔操作。
- 结尾钩子。
- 不能改变的剧情事实。
- 本次建议强化点。

规则：

- 一键重写默认基于章节摘要/特性卡片扩写和补强，而不是让模型自由重写。
- 如果用户改了章节摘要/特性卡片，需要先生成新卡片版本，再生成候选正文。
- 采用候选正文后，需要同步更新当前章节摘要、长篇记忆和后续上下文。
- 章节重新变为正常时，当前状态备注可以清空，但历史变更原因保留在版本和操作记录中。
- 在当前原型口径中，本卡片作为右侧工作区顶部常驻卡片；下面“核心信息区”描述字段完整性和业务规则。

### AI 审稿报告

展示：

- 总分。
- 分项评分。
- 评级。
- 结论摘要。
- 优点。
- 问题。
- 建议动作。
- 是否允许进入下一步。

建议分项：

- 开篇吸引力。
- 爽点密度。
- 冲突强度。
- 人设一致性。
- 情绪推进。
- 章节目标完成度。
- 结尾钩子。
- 可视频化程度。
- 平台风险。

规则：

- 审稿严格程度从策略配置读取，不能写死。
- 审稿未通过时，章节进入 `pending`，状态备注写明原因，例如“审稿分低”“人设冲突”“结尾钩子弱”。
- 审稿通过但分数一般时，可以给出优化建议，但不一定阻塞。
- 用户强制确认低分章节时，必须填写原因，原因进入操作日志和学习信号。

### 问题卡片

问题卡片用于把 AI 审稿结果转成可执行动作。

每张卡片包含：

- 问题标题。
- 通俗解释。
- 严重程度：低、中、高。
- 影响范围：仅本章、影响后文、影响视频。
- 推荐动作。
- 可选动作。
- 处理状态。

常见动作：

- AI 自动优化该问题。
- 局部重写指定段落。
- 基于摘要重写整章。
- 修改章节摘要后重写。
- 手动确认可接受。
- 返回小说详情调整大纲或设定。

规则：

- 问题卡片触发的 AI 优化只能生成候选版本。
- 多个问题可以合并生成一次优化方案，但不能静默批量覆盖正文。
- 问题处理后需要形成 `ReviewActionRecord`，记录处理前后版本、评分变化和用户是否采用。

## 候选版本流程

候选版本是章节重写、局部优化、手动编辑或问题修复后的结果。候选版本未被采用前，不影响当前正式正文。

### 候选版本状态

| 状态 | 含义 | 用户可做什么 |
| --- | --- | --- |
| 生成中 | AI 正在生成候选版本 | 查看任务进度、取消任务 |
| 待复审 | 候选版本已生成，等待 AI 审稿 | 发起复审、放弃候选 |
| 复审未通过 | 候选版本仍有明显问题 | 继续优化、放弃候选、查看差异 |
| 复审通过 | 候选版本质量可采用 | 采用、继续优化、放弃 |
| 已采用 | 候选版本成为正式版本 | 查看影响评估 |
| 已放弃 | 候选版本不再参与当前流程 | 查看历史原因 |

### 版本对比

对比内容：

- 改动摘要。
- 字数变化。
- 评分变化。
- 问题数量变化。
- 爽点和冲突变化。
- 关键事实是否变化。
- 结尾钩子是否变化。
- 风险变化。

要求：

- 默认先展示通俗改动摘要，再允许展开正文差异。
- 评分没有提升时，默认不推荐采用。
- 关键事实变化、人物关系变化、伏笔变化时，需要提示会触发影响评估。
- 版本对比按需加载，避免列表和页面首屏加载长正文。

### 采用候选版本

采用前必须展示：

- 当前正式版本。
- 即将采用的候选版本。
- 评分变化。
- 关键改动摘要。
- 是否可能影响后续章节。
- 是否影响已生成视频。

采用后系统必须：

- 将候选版本设为当前正式正文。
- 记录旧版本和新版本。
- 更新章节摘要/特性卡片。
- 更新长篇记忆。
- 触发或要求触发影响评估。
- 重新计算章节推荐动作。
- 重新计算小说列表推荐动作。

### 新改稿候选弹窗

点击“生成新改稿”后，系统先弹窗展示候选稿内容。

弹窗可编辑字段：

- 概览。
- 核心任务。
- 主要冲突。
- 爽点。
- 结尾钩子。
- 正文。

弹窗按钮：

- 重新生成：基于当前建议和用户补充要求重新生成候选。
- 保存候选：保存候选版本，但不替换当前正式正文。
- 取消：关闭弹窗，不保存本次候选。

规则：

- 保存候选用于把 AI 生成结果先留存下来，方便后续比较、继续优化或采用。
- 只有“采用候选”才会把候选变为正式正文。
- 候选保存、采用、放弃和重新生成都要记录操作来源和原因。

## 影响评估与后续章节处理

章节正文被重写、手动编辑或采用候选版本后，必须进行 AI 影响评估。影响评估判断当前改动是否破坏后续章节、长篇记忆、伏笔或视频引用。

页面默认只展示影响状态标签，例如“影响待评估”“影响未关闭”“轻微影响”。用户点击标签或采用候选后，再展开影响详情。

### 影响等级

| 等级 | 含义 | 默认处理 |
| --- | --- | --- |
| 无影响 | 只优化表达，不改变剧情事实 | 同步摘要和记忆，章节可回正常 |
| 轻微影响 | 改动了表达、情绪或局部细节，不破坏后文 | 自动同步章节摘要、特性卡片和长篇记忆 |
| 中等影响 | 改动会影响部分后续章节，但可以逐章调整 | 标记受影响章节为 `pending`，备注“受影响未调整” |
| 严重影响 | 改动破坏后续主线、人物关系、伏笔或已生成内容 | 让用户选择清空后续、逐章处理或返回修改当前候选版本 |

轻微影响和受影响未调整只作为状态备注或影响记录，不新增章节主状态。

### 中等影响处理

系统需要展示：

- 受影响章节列表。
- 每章受影响原因。
- 影响的剧情事实、人物状态、伏笔或关系。
- 推荐处理方式。

用户可选：

- AI 生成后续章节调整方案。
- 逐章进入章节详情处理。
- 批量创建候选修正任务。
- 手动确认某些章节无影响。

规则：

- 受影响章节主状态改为 `pending`。
- 处理完成前，小说不能进入完成确认。
- 批量处理只能生成候选版本或处理方案，不能直接覆盖正式正文。
- 用户手动确认无影响必须填写原因。

### 严重影响处理

系统必须给出三个选择：

1. 清空当前章之后的正文，后续章节重新生成。
2. 不清空正文，将所有受影响章节标记为 `pending` 后逐章处理。
3. 返回修改当前候选版本，让当前章尽量兼容后续章节。

规则：

- 清空后续章节属于高风险操作，必须展示章节范围、字数、已审稿情况、视频引用情况，并要求填写原因。
- 如果后续章节已经被视频引用，必须额外提示会造成视频引用异常。
- 清空操作不能删除历史版本，只取消当前正式正文引用或标记为不再当前使用。
- 严重影响未处理前，小说不能进入完成确认或待视频化。

## 章节状态规则

章节主状态仍保持简化：

| 主状态 | 含义 | 是否阻塞小说完成 |
| --- | --- | --- |
| `normal` | 当前章可继续用于后续创作 | 否 |
| `pending` | 当前章存在必须处理的问题 | 是 |
| `processing` | 当前章有进行中任务 | 是 |
| `resolved` | 问题已处理，等待复审或用户确认正常 | 默认不阻塞；严格策略可要求复审或确认 |

状态备注用于说明当前状态原因，例如：

- 轻微影响。
- 受影响未调整。
- 审稿分低。
- 正文缺失。
- 候选版本待确认。
- 视频引用风险。

规则：

- 状态备注不是独立状态。
- 当章节重新变为 `normal` 时，当前状态备注可以清空。
- 历史原因必须保存在审稿报告、影响评估、版本记录和操作日志中。
- `resolved` 不等于完全正常，页面仍要推荐“AI 复审”或“确认章节正常”；如果策略要求复审或人工确认，则通过完成门禁阻塞，而不是新增章节主状态。

## 视频引用展示规则

视频引用在章节详情中默认是状态，不独立占用一个常驻模块。

默认展示：

- 顶部状态标签：未被视频引用、已被视频引用、引用异常。
- 需要时展示引用视频数量或引用范围摘要。

展开条件：

- 当前章被视频引用。
- 当前改动可能影响视频引用。
- 用户点击视频引用状态。
- 出现引用异常。

展开后展示引用视频项目、引用版本、发布状态、异常等级和处理入口。

## 推荐动作

章节详情工作台每次只突出一个主推荐动作。

| 场景 | 主推荐动作 | 点击结果 |
| --- | --- | --- |
| 没有正文 | 生成正文 | 创建章节正文生成任务 |
| 有正文但没有摘要卡 | 生成章节摘要 | 创建摘要/特性卡片任务 |
| 有正文但没有审稿 | AI 审稿 | 创建章节审稿任务 |
| 批量正文硬失败 | 生成重写候选 | 打开硬失败原因并创建章节重写候选任务 |
| 批量正文普通风险 | 优化本章 | 打开优化建议并创建优化候选 |
| 批量正文轻微问题 | 加入润色建议 | 记录润色建议，不阻塞主流程 |
| 审稿不通过 | 优化章节 | 打开优化方案或创建重写任务 |
| 有候选版本待复审 | AI 复审候选版本 | 创建候选版本审稿任务 |
| 候选版本复审通过 | 采用候选版本 | 打开采用确认 |
| 采用后未评估影响 | 评估后续影响 | 创建影响评估任务 |
| 中等影响未处理 | 处理受影响章节 | 打开影响处理区 |
| 严重影响未决策 | 选择影响处理方式 | 打开三选一决策 |
| 问题已处理 | 确认章节正常 | 清理当前备注并刷新推荐动作 |
| 视频引用异常 | 查看视频影响 | 打开视频引用区 |

如果存在进行中任务，主动作统一变为“查看任务进度”，其他冲突动作禁用。

## 批量正文问题回流

批量正文发现章节问题后，章节详情工作台负责承接单章处理闭环。页面首屏需要回答三个问题：

- 这一章为什么有问题。
- 系统建议怎么处理。
- 处理后会影响哪些后续章节或视频引用。

处理规则：

- 硬失败章节优先展示失败原因、证据和影响范围，主动作是“生成重写候选”。
- 普通风险章节展示风险说明和优化建议，主动作是“优化本章”。
- 轻微问题章节不阻塞主流程，只记录润色建议，可在后续统一润色时处理。
- 所有正文修复都必须生成候选版本，用户确认采用后才替换当前正文。
- 采用候选版本后必须触发影响评估，不能直接关闭问题。

## 高风险操作

以下操作必须二次确认并记录原因：

- 采用候选版本导致关键事实变化。
- 清空后续章节。
- 批量创建后续章节修正任务。
- 手动确认中等或严重影响无问题。
- 强制通过低分章节。
- 修改已经被视频引用的章节。
- 放弃高评分候选版本。

确认弹窗必须说明：

- 操作对象。
- 影响章节范围。
- 是否影响小说完成。
- 是否影响视频引用。
- 是否可以回退。

## 和视频引用的关系

章节详情工作台需要展示当前章的视频引用情况：

- 未引用。
- 已创建视频但未发布。
- 已生成音频/字幕/视频。
- 已发布。
- 引用异常。

规则：

- 未发布视频可以提示“建议重新生成视频素材”。
- 已发布视频需要提示“小说修改不会自动修改平台已发布内容”。
- 已引用章节被采用新版本后，需要标记视频引用异常或待同步。
- 视频异常处理入口可以跳转到视频列表或视频详情，章节页只展示摘要和风险。

## 数据与接口边界

章节详情工作台建议按需加载，避免一次返回全部大字段。

建议接口：

- `GET /novels/:novelId/chapters/:chapterId`：获取章节详情工作台首屏数据。
- `GET /novels/:novelId/chapters/:chapterId/content`：获取当前正式正文。
- `GET /novels/:novelId/chapters/:chapterId/reviews/latest`：获取最新章节审稿。
- `GET /novels/:novelId/chapters/:chapterId/versions`：获取版本列表摘要。
- `GET /novels/:novelId/chapters/:chapterId/versions/:versionId/diff`：获取版本对比。
- `POST /novels/:novelId/chapters/:chapterId/generate`：生成章节正文。
- `POST /novels/:novelId/chapters/:chapterId/review`：发起章节审稿。
- `POST /novels/:novelId/chapters/:chapterId/rewrite`：基于摘要、问题或局部范围生成候选版本。
- `POST /novels/:novelId/chapters/:chapterId/content-versions/:versionId/adopt`：采用候选版本。
- `POST /novels/:novelId/chapters/:chapterId/versions/:versionId/discard`：放弃候选版本。
- `POST /novels/:novelId/chapters/:chapterId/impact-assessments`：发起影响评估。
- `GET /novels/:novelId/impact-cases/:impactCaseId`：查看影响案例。
- `POST /novels/:novelId/impact-cases/:impactCaseId/resolve`：首期关闭、忽略或取消影响案例。
- `POST /novels/:novelId/chapters/:chapterId/confirm-normal`：确认章节恢复正常。

接口规则：

- AI 生成、审稿、重写、影响评估都返回任务 ID，不直接等待完整结果。
- 采用候选版本、清空后续章节、确认低分通过必须做版本校验和幂等控制。
- 正文、完整审稿、完整差异内容不进入小说列表接口。
- API 响应不能返回完整提示词、完整模型响应、密钥或外部平台 token。

## 完成判断

一个章节可以视为可继续推进，至少满足：

- 有当前正式正文。
- 有当前章节摘要/特性卡片。
- 已完成章节审稿，或策略允许用户确认风险。
- 没有进行中任务。
- 没有待处理影响。
- 没有未确认候选版本阻塞。
- 视频引用风险已提示，且不阻塞小说主流程。

如果所有计划章节都满足完成判断，小说才可以继续推进到全书 AI 审稿。

## 和其他模块的关系

- 小说列表：发现待处理章节，并用推荐动作把用户带入章节详情。
- 小说详情工作台：展示章节列表、章节统计和项目级审稿，复杂单章处理跳转到章节详情。
- 任务系统：承载章节生成、审稿、重写、复审、影响评估和批量后续处理。
- 版本系统：保存正式正文、候选正文、摘要卡、审稿和影响评估历史。
- 视频系统：接收章节修改后的引用异常和同步提示。
- 学习系统：记录采用/放弃、评分变化、强制通过原因和归档原因，后续用于系统自我成长。

## 原型设计要求

后续画原型时，章节详情工作台至少需要覆盖：

- 顶部状态和主推荐动作。
- 左侧章节导航。
- 当前正文阅读区。
- 章节摘要/特性卡片。
- 审稿问题卡片。
- 一键重写/局部优化入口。
- 候选版本确认和版本对比。
- 影响评估三种处理方式。
- 视频引用提示。
- 任务进度、失败原因和重试入口。

首期研发可以裁剪成轻量版本，但需求和原型必须保留完整形态，避免后续小说质量调试没有承载页面。
