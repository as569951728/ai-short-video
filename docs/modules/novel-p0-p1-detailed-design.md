# 小说系统 P0 / P1 细化设计

本文档用于收口小说系统后续需求设计优先级。P0 负责主链路闭环，P1 负责质量、体验和可维护性增强。研发派包和测试验收时，以本文档作为优先级判断的总入口，再回到具体模块文档查看细节。

## 优先级定义

| 优先级 | 含义 |
| --- | --- |
| P0 | 不设计清楚会卡住小说主链路闭环，或影响近期研发任务包 |
| P1 | 不影响主链路继续走，但会影响质量、体验或测试可靠性 |

## P0 主链路闭环

P0 目标：让小说从试写、批量正文、全书审稿，到完成并可被视频系统引用，形成完整闭环。

| 模块 | 细化设计 | 主要文档 |
| --- | --- | --- |
| 试写闭环 | 默认生成 3 个完整第 1 章候选；AI 推荐排第一高亮但不自动选择；候选评分必须有证据、扣分点和策略版本；用户选择后作为 `selected_for_trial`，生成第 2-3 章；第 2 章硬失败暂停，第 3 章不继续生成；试写总评决定是否进入批量正文。 | `docs/modules/novel-trial-writing-loop.md` |
| 第 1 章候选对比 | 顶部 AI 推荐结论卡；三张候选卡展示综合分、开篇策略、首句、300 字摘要、结尾钩子、风险、评分证据；全文用抽屉阅读；操作包括“选这个继续试写”“基于此版优化”“重新生成 3 个”。 | `docs/modules/novel-trial-writing-loop.md` |
| 试写总评页 | 不是报告页，而是决策页。顶部给结论：进入批量正文、先优化或返回上游。中部展示前三章表现、核心问题、关键维度评分、风险说明。底部给主动作：确认进入批量正文、优化章节、返回章节目录、返回设定。 | `docs/modules/novel-trial-writing-loop.md` |
| 正文生成策略快照 | 试写通过或风险继续后生成 `BodyGenerationStrategySnapshot`。包含第 1 章基础版本、前三章总结、主角写法、爽点节奏、冲突写法、文风规则、章节结尾规则、长篇记忆、风险项、加强复审规则。版本化，不静默覆盖。 | `docs/modules/novel-trial-writing-loop.md` |
| 批量正文生成 | 默认 5 章一批；每章生成正文、章节特性卡、单章审稿；每批结束生成批次总结；风险继续时前 10 章强制逐章严格审稿；硬失败暂停当前批次，普通风险继续但记录。 | `docs/modules/novel-task-package-6-detailed-design.md`、`docs/modules/novel-batch-body-generation.md` |
| 章节详情工作台 | 单章问题处理中心。首屏解释“为什么有问题、建议怎么处理、会影响哪里”。硬失败主动作是“生成重写候选”；普通风险是“优化本章”；轻微问题是“加入润色建议”；所有修复先生成候选，用户采用后才替换正文。 | `docs/modules/novel-chapter-workbench.md` |
| 影响评估闭环 | 采用章节候选后必须触发。影响分为无影响、轻微、中等、严重。无影响直接关闭；轻微写入长篇记忆；中等标记受影响章节并生成修正建议；严重影响三选一：清空后续章节、逐章修正、回改当前候选。 | `docs/modules/chapter-rewrite-impact-handling.md`、`docs/modules/novel-chapter-workbench.md` |
| 全书审稿 | 批量正文完成后启动。输入包括所有正文、章节卡、审稿报告、长篇记忆、影响案例、策略快照。输出总分、问题列表、章节风险、伏笔回收、爽点衰减、AI 味、视频化潜力。中高风险未处理不能完成。 | `docs/modules/novel-full-review-gate.md` |
| 完成确认 | 全书审稿通过后，用户确认小说完成。弹窗展示总分、遗留风险、影响案例状态、正文版本范围、视频化风险。确认后写完成决策记录，冻结完成快照。完成确认不等于待视频化。 | `docs/modules/novel-full-review-gate.md`、`docs/modules/novel-state-machine.md` |
| 待视频化确认 | 完成后做视频化检查。检查首条视频钩子、推荐切片、旁白顺畅度、敏感风险、章节引用完整性。通过后生成 `VideoReadinessSnapshot`，小说进入 `video_ready`。 | `docs/modules/novel-video-readiness.md` |
| 视频引用状态 | 小说模块只展示引用状态，不创建视频。状态为未准备、可被引用、已引用、引用异常。小说修改后，如果影响已引用章节，生成引用异常，提示去视频系统处理。 | `docs/modules/video-reference-exceptions.md`、`docs/modules/video-system.md` |

### P0 验收底线

- 没有试写通过或风险确认，不能批量正文。
- 没有正文生成策略快照，不能批量正文。
- 批量正文部分失败时，已完成章节必须保留。
- 中等或严重影响未关闭，不能全书审稿。
- 全书审稿过期，不能确认完成。
- 完成确认和待视频化确认必须分开。
- 小说列表不能出现“创建视频”主动作。

## P1 质量与体验增强

P1 目标：让 P0 主链路更可靠、更好用、更方便后续优化，但不阻塞主链路推进。

### P1 分层原则

P1 不等于“以后再想”。它分为三层：

| 层级 | 含义 | 推进方式 |
| --- | --- | --- |
| P1a 即时修正 | 已经影响用户理解或测试可靠性的小问题 | 穿插进最近任务包修正，不单独开大包 |
| P1b 伴随增强 | 和 P0 主链路强相关，但不改变主流程门禁 | 跟随任务包 5-8 分段落地 |
| P1c 质量运营 | 需要积累数据后才有价值的复盘、调参和策略优化 | P0 主链路稳定后集中做 |

P1 的产品口径是“增强判断、增强记录、增强解释”，不是提前做自动学习、复杂运营后台或真实视频生产。

| 模块 | 细化设计 | 建议阶段 |
| --- | --- | --- |
| 评分复盘面板 | 展示 AI 推荐采纳率、高分候选试写失败率、低分强行继续结果、各题材评分稳定性、常见误判维度。第一期只记录和查看，不自动学习。 | P0 主链路稳定后做轻量版 |
| 评分策略调参 | 策略版本化：权重、阈值、硬门槛、扣分规则、题材适配、提示词版本。调参必须发布新版本，旧结果保留旧版本引用。 | P0 主链路稳定后 |
| 任务中心体验优化 | 修正 stale 任务仍提示可重试的问题；任务详情展示对象、来源版本、失败原因、下一步、是否可换模型重试；失败任务支持“重试 / 换模型重试 / 放弃结果”。 | 穿插任务包 5/6 |
| 批次总结增强 | 每批展示风险趋势、爽点衰减、人物稳定性、AI 味变化、下一批注意事项。用于帮助用户判断是否继续下一批。 | 任务包 6 后 |
| 候选对比增强 | 章节当前版 vs 候选版摘要对比，不做复杂 diff；展示改动收益、风险、影响范围、是否建议采用。 | 任务包 6 后 |
| 首条测试卡 | 试写通过后可生成，不阻塞批量正文。包含标题、前 3 秒旁白、首屏字幕、核心冲突、视频钩子、风险提示。用于提前验证视频化潜力。 | 试写通过后，可选 |
| 成本与耗时提示 | 批量生成、重试、重新生成候选、全书审稿前展示预计耗时、章节数、可能成本、取消后果。首期做轻量提示，不做精确计费。 | 可穿插 |
| 内容安全提示 | 在方向、试写、全书审稿、待视频化前做轻量风险提示。严重内容安全风险阻断；普通平台风险提示并记录。 | 全链路轻量提示 |
| 原型一致性复盘 | 每完成 2 个任务包，检查列表、详情、章节页、任务抽屉的状态文案、推荐动作、按钮边界是否一致，避免“按钮漂移”。 | 每两个任务包复盘 |

### P1a：即时修正

#### 任务中心 stale 重试提示修正

背景：任务包 4 测试发现，sourceVersionRefs 已过期的失败任务在详情中仍可能展示“可重试 / 重试任务”，但真实 retry 会被 `CANDIDATE_STALE` 阻断。这不会破坏资产安全，但会误导用户。

设计：

- 任务详情的 `canRetry` 必须同时读取任务状态、失败类型、sourceVersionRefs 当前有效性和冲突范围。
- 如果上游版本已变化，任务详情不显示“重试任务”，改为展示“上游内容已经变化，旧任务不能直接重试”。
- 下一步动作改为：
  - 查看当前最新候选或当前资产。
  - 基于最新版本重新发起生成。
  - 放弃旧任务结果。
- 任务事件中保留旧任务失败和被阻断原因，便于测试和复盘。

验收：

- stale 任务详情不出现可点击的重试主按钮。
- 调用 retry 接口仍返回统一错误结构，错误码为 `CANDIDATE_STALE` 或等价契约错误。
- 页面能解释为什么不能重试，以及用户下一步应该去哪。

不做：

- 不在本项里做复杂任务大屏。
- 不自动把旧任务结果迁移到新版本。

#### P1 轻量提示统一口径

任务包 5 开始会进入更高成本的试写、重试、审稿和候选比较，因此需要先统一提示口径：

- 预计耗时：只展示“大约需要几分钟 / 可能较久”，不展示复杂 token。
- 成本风险：只在重新生成 3 个第 1 章候选、批量正文、全书审稿、多次重试时提示。
- 安全风险：展示风险等级、原因和推荐动作，不默认展示敏感原文。
- 评分风险：展示“为什么不推荐继续”，允许用户在非阻塞档位填写原因继续。

验收：

- 小白默认只看到可理解提示。
- 高级信息折叠，不抢主流程。
- 高风险阻断必须清楚说明原因。

### P1b：伴随 P0 的增强能力

#### 评分记录和评分证据

评分不是单次 UI 文案，而是后续质量调试的基础数据。任务包 5 先做记录地基，后续再做复盘面板。

适用对象：

- 第 1 章候选。
- 第 2-3 章试写结果。
- 单章正文审稿。
- 批次总结。
- 全书审稿。
- 待视频化检查。

每次评分需要记录：

- 被评分对象：资产类型、资产 ID、版本 ID、章节 ID 或任务 ID。
- `scoringStrategyVersion`。
- 维度分、权重、综合分和门禁结论。
- 证据说明和扣分点。
- 关键风险，例如开篇弱、主角弱、爽点不足、AI 味、平台风险。
- AI 推荐动作。
- 用户动作：采纳、未采纳、强制继续、返回上游、重新生成。
- 后续结果：第 2-3 章表现、试写总评、是否进入批量正文。

前端展示：

- 默认显示综合分、门禁结论和 1-3 条关键理由。
- 展开后显示维度分、证据、扣分点和策略版本。
- AI 推荐候选排第一并高亮，但不自动选择。

验收：

- 每个第 1 章候选都有评分版本、维度分、证据、扣分点和门禁结论。
- 用户没有选择 AI 推荐时，需要记录“未采纳推荐”的事实，可选记录原因。
- 低分或风险继续必须填写原因。

#### 候选对比增强

候选对比用于帮助用户判断“这个候选值不值得采用”，不是复杂代码 diff。

适用场景：

- 第 1 章候选对比。
- 章节重写候选对比。
- 章节摘要/特性卡候选对比。

首期展示：

- 当前版本摘要。
- 候选版本摘要。
- 改动收益：钩子更强、冲突更清晰、AI 味更低、人物更稳等。
- 新增风险：改变关键事实、影响后文、平台风险、视频引用风险等。
- 影响范围：仅本章、影响后续章节、影响已视频引用内容。
- AI 建议：推荐采用、建议继续优化、不建议采用。

验收：

- 用户采用候选前能看到“改了什么、好处是什么、风险是什么”。
- 不要求首期做逐字文本 diff。
- 高风险候选采用必须触发二次确认和影响评估。

#### 批次总结增强

P0 要求每批正文生成后有批次总结。P1 增强总结的可决策性。

增强字段：

- 本批章节范围和完成状态。
- 每章评分趋势。
- 爽点兑现趋势。
- 主角稳定性。
- 关键配角稳定性。
- AI 味变化。
- 平台和视频化风险变化。
- 和上一批相比的风险变化。
- 下一批必须注意的 3 条事项。

前端展示：

- 顶部一句话结论：建议继续、建议先优化、必须处理失败章节。
- 风险趋势用简单标签，不展示复杂图表。
- 可从总结直接进入风险章节详情。

验收：

- 批次总结能支持用户判断“继续下一批还是先修章”。
- 普通低分继续生成后，风险必须在批次总结里看得到。
- 批次总结会进入下一批生成输入，避免每批互相断档。

#### 首条测试卡

首条测试卡用于在全书完成前验证开篇视频化潜力，但不能让用户误以为小说已经正式待视频化。

出现时机：

- 试写总评 `pass` 或 `pass_with_suggestions` 后。
- 用户风险继续进入批量正文时也可生成，但必须带风险提示。

内容：

- 推荐测试章节范围。
- 标题钩子 2-3 个。
- 前 3 秒旁白 2-3 个。
- 首屏字幕 2-3 个。
- 核心冲突。
- 结尾悬念。
- 视频化风险。
- 建议平台或内容形态。

规则：

- 首条测试卡不阻塞批量正文。
- 首条测试卡不是 `VideoReadinessSnapshot`。
- 生成测试视频属于后续视频系统或运营实验，不进入任务包 5。
- 如果用户手动记录测试结果，进入运营实验记录和系统自我成长信号。

验收：

- 试写通过后可以看到“生成测试卡”的可选入口。
- 用户跳过测试卡仍可进入批量正文。
- 页面文案明确“测试卡不是正式视频化”。

#### 成本与耗时提示

成本提示要帮助用户做决定，不能把小白吓住。

提示场景：

- 重新生成多个第 1 章候选。
- 生成第 2-3 章试写。
- 批量正文生成。
- 全书审稿。
- 多次重试或换模型重试。

提示分级：

- 普通：展示预计耗时。
- 提醒：抽屉或确认文案提示可能耗时较久。
- 警告：高成本任务或多次重试需要确认。
- 阻塞：模型不可用、额度不足、并发已满、文本超过系统上限。

验收：

- 批量任务开始前能看到章节数、预计耗时和取消后果。
- 重试前提示可能重新消耗额度。
- 小白默认不展示 token、单价和完整成本公式。

#### 内容安全和平台风险提示

内容安全风险和普通质量分分开处理。

检查入口：

- 方向生成后。
- 设定、大纲和章节目录采用前。
- 章节正文生成后。
- 全书审稿时。
- 待视频化检查时。

风险等级：

- `info`：不阻塞，只记录和提示。
- `warning`：建议修改，允许用户记录原因后继续。
- `blocking`：阻塞下一步，不能通过接受风险绕过。

验收：

- 严重内容安全风险不能被高质量分覆盖。
- 平台风险有通俗说明和推荐动作。
- 默认页面不展示过多敏感原文。

### P1c：质量运营和策略调试

#### 评分复盘面板

评分复盘面板在 P0 主链路稳定、有足够评分和用户选择数据后再做。它服务管理员或高级用户，不进入小白主流程。

核心指标：

- AI 推荐采纳率。
- AI 推荐候选试写通过率。
- 高分候选失败率。
- 低分强制继续成功率。
- 不同题材评分稳定性。
- 常见扣分维度排行。
- 常见用户未采纳原因。
- 不同模型或提示词版本的评分表现。

页面目标：

- 找出评分误判。
- 找出某类题材默认权重是否不合适。
- 找出提示词、模型或策略版本是否导致系统性低分。

验收：

- 面板只做复盘，不自动改权重。
- 每个结论都能点回样本证据。
- 样本不足时显示“数据不足”，不强行给结论。

#### 评分策略调参

调参必须版本化，不能直接改历史结果。

策略内容：

- 维度权重。
- 综合分通过/风险/阻塞阈值。
- 关键维度硬门槛。
- 扣分规则。
- 题材适配规则。
- 审稿提示词版本。
- 适用阶段：方向、试写、章节、全书、视频化。

发布流程：

1. 创建策略草稿。
2. 用历史样本回放，查看新旧策略结论差异。
3. 人工确认发布。
4. 新评分引用新版本。
5. 旧评分保留旧版本，不被重算覆盖。
6. 需要回滚时发布回滚版本，而不是删除历史版本。

验收：

- 不允许直接修改已发布策略。
- 策略发布后只影响新任务。
- 策略回放必须展示差异样本，避免盲调。

#### 原型一致性复盘

每完成 2 个研发任务包做一次原型和文案复盘，防止状态、按钮和推荐动作漂移。

复盘范围：

- 小说列表。
- 小说详情工作台。
- 章节详情工作台。
- 任务抽屉。
- 试写总评。
- 批次总结。
- 全书审稿。
- 待视频化页。

检查项：

- 同一个状态是否使用同一个标签和颜色。
- 同一个风险是否使用同一套等级。
- 主按钮是否符合当前推荐动作。
- 禁用按钮是否说明原因。
- 高风险操作是否二次确认。
- 任务失败是否告诉用户下一步。

输出：

- 文案和状态问题清单。
- 需要修正的 P1a 小项。
- 是否影响下一任务包派发的结论。

### P1 数据口径

首期可以复用现有 `ReviewReport`、`GenerationTask`、`AssetDecisionRecord`、`OperationLog` 和 metadata 字段承载 P1 数据；当数据查询和复盘变复杂时，再拆独立表。不要为了 P1 先上复杂分析库。

建议逐步沉淀的对象：

| 对象 | 用途 |
| --- | --- |
| `QualityScoreRecord` | 保存评分版本、维度分、证据、扣分点和门禁结论 |
| `QualityFeedbackSignal` | 保存用户选择、未采纳原因、强制继续原因和后续结果 |
| `ScoringStrategyVersion` | 保存评分权重、阈值、硬门槛和提示词版本 |
| `TaskCostSnapshot` | 保存任务耗时、token、成本和失败重试次数摘要 |
| `SafetyRiskFinding` | 保存内容安全、平台、原创化和版权风险摘要 |
| `TestCard` | 保存首条测试卡内容 |
| `OperationExperimentRecord` | 保存测试卡或测试视频的手动运营结果 |

### P1 不做清单

- 不做自动学习和自动调权重。
- 不做真实模型训练。
- 不做复杂 BI 数据平台。
- 不做正式视频生成和发布。
- 不做逐字级复杂正文 diff。
- 不把成本、模型参数、完整提示词和完整模型响应暴露给小白用户。
- 不允许 P1 功能绕过 P0 的版本、任务、审稿和影响评估门禁。

### P1 验收底线

- P1 不应拖慢 P0 主链路。
- 所有调参和复盘都必须可追溯版本。
- 任务失败提示必须告诉用户下一步能做什么。
- 成本、安全提示不能吓用户，但高风险必须阻断。
- 首条测试卡不能让用户误以为小说已经正式视频化。

## 推荐推进顺序

1. 先完成 P0 任务包 5：试写闭环和章节详情工作台。
2. 再完成 P0 任务包 6：批量正文、章节重写和影响评估。
3. 再完成 P0 任务包 7：全书审稿、完成确认和待视频化。
4. 再完成 P0 任务包 8：视频引用快照和引用异常。
5. P1 先穿插做任务中心体验、评分记录、成本提示这三个轻量点。
6. P1 的评分复盘面板、策略调参和批次总结增强，在主链路稳定后集中设计和研发。

## 任务包 5 派发前复盘口径

任务包 5 派发前，需求主控需要确认：

- P0 主方向仍然是“先把小说生成质量做好”，不是提前转向视频生产。
- 任务包 5 只做试写闭环、章节详情工作台首版和 P1a 任务中心 stale 提示修正。
- 任务包 5 需要落下 P1 数据地基：评分版本、评分证据、用户选择反馈、试写结果回流。
- 任务包 5 不做评分复盘面板、不做自动调权重、不做批量正文、不做正式视频化。
- 测试会话需要同时验收 P0 主流程和 P1 数据记录是否可追溯。
