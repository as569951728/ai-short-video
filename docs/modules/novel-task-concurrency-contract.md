# 任务并发与批量任务契约

本文档补齐 `GAP-P1-011`：统一小说系统中 AI 长任务、批量任务、父子任务、并发互斥、重试、取消、部分成功和上游版本变化的需求口径，避免重复生成、旧上下文写入新版本、任务成功但业务状态不清。

本文档只定义需求级契约，不讨论具体队列、锁或 worker 实现方案。

## 核心结论

任务系统必须解决三个问题：

- 同一对象不能同时执行互相冲突的任务。
- 批量任务必须能表达部分成功、部分失败和可继续处理。
- 任务执行中依赖的上游版本变化时，旧任务结果不能静默写入当前资产。

所有会改变创作资产、审稿报告、长篇记忆、影响案例、视频快照的任务，都必须绑定业务对象和输入版本摘要。

## 任务绑定契约

每个任务必须至少绑定：

| 字段 | 需求说明 |
| --- | --- |
| `tenantId` | 后续多租户隔离 |
| `novelId` | 所属小说 |
| `objectType` | 绑定对象类型，例如方向、设定、大纲、章节、视频 |
| `objectId` | 绑定对象 ID |
| `taskType` | 明确业务任务类型，不能只叫 generate |
| `parentTaskId` | 批量任务或流程任务的父任务 |
| `batchId` | 同一批次任务标识 |
| `sourceVersionRefs` | 任务创建时读取的关键版本引用摘要 |
| `conflictScope` | 任务互斥范围，例如小说级、阶段级、章节级、版本级、视频引用级 |
| `conflictKey` | 同一互斥范围下的唯一冲突键，用于判断是否已有冲突任务 |
| `triggerSource` | 用户手动、系统自动、重试、批量任务 |
| `idempotencyToken` | 防重复创建的业务幂等键 |
| `requestHash` | 请求参数摘要，用于识别同一幂等键下参数是否冲突 |

任务不能只绑定小说，不绑定具体对象。否则后续无法判断冲突范围、版本过期和结果归属。

`conflictScope` 和 `conflictKey` 是互斥判断的事实口径。后续具体实现可以用数据库唯一约束、事务锁、队列锁或 Redis 锁，但业务语义必须保持一致：

- 同一 `conflictScope + conflictKey` 下，只能存在一个会改变事实源的活跃任务。
- 只读审稿、候选复审等任务可以使用更小的冲突范围，避免把整本小说锁死。
- 创建任务接口必须在返回前完成冲突检查，不能只依赖 worker 启动后再判断。
- worker 启动时仍要再次校验冲突，防止并发请求绕过接口层。

## 幂等契约

以下动作必须具备业务级幂等口径：

- 创建方向生成任务。
- 创建设定、大纲、章节目录生成任务。
- 创建试写任务。
- 创建批量正文生成任务。
- 创建章节重写或局部优化任务。
- 创建章节审稿、全书审稿、视频化检查任务。
- 重试任务。
- 取消任务。
- 确认采用任务结果。
- 放弃任务结果。
- 清空后续章节。
- 确认小说完成。
- 确认待视频化。
- 创建视频项目。

幂等语义：

- 用户重复点击同一主按钮，不能创建多个等价任务。
- 网络重试同一请求，不能重复切换版本或重复写决策记录。
- 如果同一幂等键对应动作已完成，应返回已有结果摘要。
- 如果同一幂等键对应动作正在处理中，应返回当前任务摘要。
- 如果请求参数和已有幂等记录冲突，应返回可解释错误。

## 冲突范围定义

任务互斥不能只按小说整体锁死，也不能完全放开。互斥范围分为：

| 范围 | 说明 | 示例 |
| --- | --- | --- |
| 小说级 | 会影响全书主链路或多个阶段 | 批量正文、全书审稿、视频化检查 |
| 阶段级 | 影响某一阶段结构资产 | 全书大纲、阶段大纲、章节目录 |
| 章节级 | 只影响单章正文、摘要或审稿 | 章节生成、章节重写、章节审稿 |
| 版本级 | 只处理某个候选版本 | 候选复审、格式修复 |
| 视频引用级 | 影响某个视频项目或引用范围 | TTS、字幕、渲染、引用复查 |

互斥范围越小，系统并发能力越好；但只要可能改变同一事实源，就必须互斥。

## 批量任务语义

批量任务必须拆成父任务和子任务。

父任务负责：

- 批次范围。
- 总体进度。
- 子任务汇总。
- 部分成功/失败状态。
- 是否阻塞主流程。
- 用户下一步建议。

子任务负责：

- 单章、单阶段、单视频步骤等具体执行。
- 单独失败原因。
- 单独重试。
- 单独结果版本或报告。

## 批量任务状态

父任务仍使用统一任务主状态，但需要额外汇总字段：

| 汇总字段 | 说明 |
| --- | --- |
| `totalCount` | 总子任务数 |
| `completedCount` | 已完成数 |
| `failedCount` | 失败数 |
| `cancelledCount` | 取消数 |
| `waitingConfirmationCount` | 待确认数 |
| `currentItemLabel` | 当前正在处理的章节或步骤 |
| `partialSuccess` | 是否已产生部分可用结果 |
| `blocksMainFlow` | 是否阻塞主流程进入下一阶段 |

批量任务结果语义：

| 场景 | 父任务状态 | 主流程影响 |
| --- | --- | --- |
| 全部子任务成功且无待确认 | `completed` | 可进入下一步 |
| 部分子任务成功、部分失败 | `failed` 或 `waiting_confirmation` | 主流程通常阻塞，用户处理失败项 |
| 子任务有待确认候选 | `waiting_confirmation` | 等待用户采用、放弃或继续优化 |
| 用户取消未开始子任务 | `cancelled` 或 `failed` | 已完成结果保留，主流程按缺失项判断 |
| 上游版本变化导致批次失效 | `cancelled` | 基于新版本重新创建批次 |

如果批量正文生成部分成功，已生成章节正文和审稿结果必须保留；未生成章节显示为缺失或失败，小说不能进入全书审稿。

## 章节批量生成规则

章节批量生成特别容易出错，必须遵守：

- 按章节顺序或按阶段分段顺序推进。
- 后一章生成前，需要读取前一章当前正文、章节摘要和长篇记忆。
- 如果某章失败，后续依赖该章记忆的章节默认暂停。
- 如果策略允许跳过失败章节继续生成，必须标记后续章节存在上下文风险。
- 每章完成后需要保存正文、摘要、审稿或审稿任务引用。
- 批量任务完成前不能把小说直接标记为正文全部完成。

## 上游版本变化处理

每个任务创建时必须记录 `sourceVersionRefs`，至少包含和任务相关的关键当前版本。

任务执行中如果上游版本变化：

| 任务状态 | 处理 |
| --- | --- |
| `queued` | 取消任务，原因写“前置内容已变化” |
| `processing` | 在安全点取消；如果已产生结果，标记为基于旧版本的候选 |
| `waiting_confirmation` | 候选标记为过期，默认不推荐采用 |
| `completed` | 保留历史，但如果结果影响当前流程，需要重新计算过期状态 |
| `failed` | 保留失败历史，重试时必须基于新版本重新创建 |

安全点示例：

- 调用模型前。
- 解析输出前。
- 写候选版本前。
- 写正式版本前。
- 写视频引用快照前。

任务结果写入前必须对比 `sourceVersionRefs` 和当前事实源：

- 如果上游关键版本完全一致，可以按正常流程写入候选或报告。
- 如果上游轻微变化，只能写为带 `soft_stale` 或 `risk_stale` 的候选，不能静默推荐采用。
- 如果上游关键版本变化，任务结果不能成为当前正式资产，只能取消或保存为 `hard_stale` 历史候选。
- 批量任务中的单个子任务也要独立校验版本，不能因为父任务未过期就跳过子任务校验。

如果任务已经进入“写正式版本”阶段，需求上要求业务动作具备原子口径，避免保存一半。

## 失败是否阻塞主流程

不是所有失败任务都阻塞小说推进。

| 失败任务 | 是否阻塞 | 说明 |
| --- | --- | --- |
| 当前阶段主生成任务失败 | 是 | 例如设定生成、大纲生成 |
| 当前阶段审稿任务失败 | 通常是 | 策略要求审稿时必须处理 |
| 可选优化任务失败 | 否 | 失败后可回到当前版本继续 |
| 候选复审失败 | 阻塞该候选采用 | 不阻塞当前正式版本 |
| 章节生成失败 | 是 | 对该章节和全书审稿阻塞 |
| 单章审稿失败 | 按策略 | 可重试，必要时阻塞章节完成 |
| 影响评估失败 | 是 | 不能确认章节无影响 |
| 长篇记忆同步失败 | 通常是 | 后续章节生成可能不可信 |
| 全书审稿失败 | 是 | 不能确认完成 |
| 视频化检查失败 | 是 | 不能创建视频项目 |
| 非引用范围视频任务失败 | 否 | 不影响小说创作主链路 |

## 重试契约

重试不是把失败任务改回 `queued`。

规则：

- 重试创建新任务。
- 新任务通过 `retryOfTaskId` 关联原任务。
- 原失败任务保留失败历史。
- 重试时必须重新校验上游版本。
- 如果上游版本变化，默认基于新版本创建任务，不沿用旧输入。
- 高成本任务重试前需要提示成本风险。
- 多次失败后，推荐动作可以从“重试”变为“调整输入或换模型”。

## 取消契约

用户取消任务后：

- 已完成子任务结果保留。
- 未开始子任务取消。
- 处理中子任务在安全点取消。
- 已生成候选但未采用的结果保留为历史候选或标记取消。
- 释放当前阶段的处理中展示。
- 重新计算推荐动作。

以下任务取消需要额外提示：

- 批量正文生成：可能留下部分章节正文。
- 全书审稿：完成确认不可继续。
- 影响评估：章节采用后的影响未关闭。
- 视频渲染：已生成的音频或字幕不自动删除。

## 自动任务触发规则

某些任务可以由系统自动触发，但必须可追踪：

| 触发事件 | 自动任务 | 规则 |
| --- | --- | --- |
| 章节首次正文生成完成 | 章节审稿、长篇记忆同步 | 可自动创建，但要显示进度 |
| 章节候选被采用 | 章节复审、影响评估、长篇记忆同步 | 必须创建或提示任务 |
| 全部章节完成 | 全书审稿 | 可以推荐，不应静默执行，除非策略允许 |
| 完成确认后 | 视频化检查 | 可以自动创建检查任务 |
| 视频引用章节变化 | 视频引用复查 | 可以自动创建异常检查 |

自动任务失败时，推荐动作必须能引导用户处理，不能把失败隐藏在后台。

## 用户可见反馈

任务并发和批量任务的复杂性不能直接暴露给小白，但用户必须知道：

- 系统正在处理哪一步。
- 当前处理到第几章或第几个步骤。
- 哪些成功了，哪些失败了。
- 是否可以继续。
- 如果不能继续，下一步点什么。

批量正文生成示例文案：

- “已生成 12/60 章，第 13 章生成失败。”
- “前 12 章已保留，处理失败章节后可以继续。”
- “第 13 章失败会影响后续章节，建议先重试或调整章节卡片。”

## 接口行为契约

任务创建类接口必须：

- 校验业务前置条件。
- 校验冲突任务。
- 生成或复用幂等任务。
- 记录输入版本摘要。
- 记录 `conflictScope`、`conflictKey`、`idempotencyToken` 和 `requestHash`。
- 返回任务 ID 和状态摘要。

幂等返回规则：

- `idempotencyToken` 相同且 `requestHash` 相同：返回已有任务或已有动作结果。
- `idempotencyToken` 相同但 `requestHash` 不同：返回参数冲突错误，不创建新任务。
- `idempotencyToken` 不同但 `conflictScope + conflictKey` 冲突：返回冲突任务摘要，引导用户查看进度或取消。

任务结果确认类接口必须：

- 校验任务仍处于可确认状态。
- 校验任务结果基于的上游版本仍有效。
- 调用资产采用与过期副作用规则。
- 返回新的状态摘要和推荐动作。

任务取消类接口必须：

- 判断是否可取消。
- 记录取消原因。
- 处理父子任务状态。
- 返回新的状态摘要和推荐动作。

## 验收口径

- 重复点击同一生成按钮不会创建多个等价任务。
- 同一章节不会同时出现两个会改正文的任务。
- 章节重写候选未采用前，不影响当前正文和长篇记忆。
- 批量正文部分失败时，已成功章节保留，失败章节有明确下一步。
- 上游版本变化后，旧任务候选默认不可直接采用。
- 全书审稿期间不能同时批量重写正文。
- 视频化检查期间，如果章节正文变化，检查结果会过期或取消。
- 任务失败能说明是否阻塞主流程。
- 任务取消后页面不会停留在生成中。
