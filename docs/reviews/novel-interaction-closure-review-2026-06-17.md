# 小说系统交互闭环评审记录

本文档用于按业务链路评审当前原型交互是否能支撑小说系统闭环。评审目标不是一次性重做所有页面，而是每次只审一段业务，发现缺口后回写到需求和原型文档。

## 评审方法

每条业务链路按以下口径判断：

| 维度 | 评审问题 |
| --- | --- |
| 业务目标 | 这条链路最终要让用户完成什么 |
| 页面入口 | 用户从哪里开始 |
| 页面状态 | 用户能不能看懂当前做到哪一步 |
| 用户动作 | 用户下一步能点什么，是否足够少 |
| 系统结果 | 点击后产生任务、候选、正式版本还是状态变化 |
| 异常处理 | 失败、低分、待确认、中途离开时能不能继续 |
| 回退路径 | 用户能不能返回上一步或放弃当前结果 |
| 下一步承接 | 当前链路结束后，下一条链路在哪里接住 |

闭环结论分三类：

| 结论 | 含义 |
| --- | --- |
| 闭环 | 主路径、异常、回退和下一步都明确 |
| 部分闭环 | 主路径成立，但仍有状态、按钮、异常或文档冲突 |
| 未闭环 | 用户完成不了业务目标，或结果没有下一步承接 |

## 业务链路拆分

| 序号 | 链路 | 主要页面 | 评审顺序 |
| --- | --- | --- | --- |
| 1 | 创建小说到待生成设定 | 小说列表、创建小说向导、小说详情 | 本轮评审 |
| 2 | 生成设定与确认设定 | 小说详情、任务抽屉、结果确认抽屉 | 下一轮 |
| 3 | 生成全书大纲、阶段大纲与章节目录 | 小说详情、大纲区、目录区 | 待评审 |
| 4 | 试写章节与试写调试 | 小说详情、章节详情、试写结果抽屉 | 待评审 |
| 5 | 批量生成正文与任务进度 | 小说详情章节管理区、任务抽屉 | 待评审 |
| 6 | 单章正文查看、编辑、重写和候选采用 | 章节详情工作台 | 待评审 |
| 7 | 章节重写影响后文处理 | 章节详情、影响处理抽屉、小说详情 | 待评审 |
| 8 | 全书审稿、优化和完成确认 | 小说详情、全书审稿区、完成确认弹窗 | 待评审 |
| 9 | 待视频化确认与视频引用状态 | 小说详情、视频列表、小说列表 | 待评审 |

## 链路 1：创建小说到待生成设定

### 业务目标

让完全不懂小说创作的用户，从小说列表点击“创建小说”，在系统推荐或少量手动输入下选定一个小说方向，并创建正式小说项目。创建完成后，小说进入“待生成设定”，由小说详情页承接下一步。

### 当前页面承接

| 环节 | 当前设计 | 结论 |
| --- | --- | --- |
| 入口 | 小说列表表格操作区有“创建小说” | 闭环 |
| 来源偏好 | 创建向导支持系统推荐、引用热点、手动想法 | 闭环 |
| 高级设置 | 高级设置折叠，章节数和每章上限为数字输入 | 闭环 |
| 生成方向 | 默认生成 3 个方向，有进度、失败和重试 | 闭环 |
| 选择方向 | 方向卡可使用、编辑、融合、重新生成 | 部分闭环 |
| 确认创建 | 确认页展示标题、方向、题材、长度和下一步 | 闭环 |
| 创建后承接 | 回到列表并高亮新行；进入详情后生成设定 | 部分闭环 |

### 已经闭环的点

- 小白用户可以不填专业内容，直接用系统推荐生成方向。
- 生成方向是任务化的，有进度、失败原因和重试入口。
- 每张方向卡能编辑，避免用户只能被动接受 AI 的结果。
- 确认创建前会汇总标题、方向、题材、目标章节数和每章上限。
- 正式小说项目从“待生成设定”开始，避免创建一个空项目让用户迷路。

### 发现的缺口

| 缺口 | 影响 | 本轮处理 |
| --- | --- | --- |
| 创建完成后的承接文案冲突：有的文档说列表主按钮是“生成小说设定” | 违背列表只显示“详情”的收敛规则 | 已要求改为“列表主按钮为详情，详情页推荐生成小说设定” |
| 方向卡编辑后是否重新评分、重新标风险不明确 | 用户编辑核心卖点后，原 AI 推荐标记可能失真 | 已补充：核心字段变化后需刷新评分/风险或标记评分待更新 |
| 草稿恢复入口偏弱 | 用户中途离开后，可能不知道从哪里继续 | 后续在创建向导链路补充“创建未完成”入口展示规则 |
| 内容安全风险只写在异常表，确认创建页的阻断方式还不够显性 | 有风险方向可能被误创建 | 后续在创建确认页补充风险阻断状态和调整入口 |

### 本轮结论

链路 1 为“部分闭环”。

主路径已经成立：列表进入创建页，创建页完成方向确认，创建后回列表，再由详情页承接“生成设定”。需要修正的关键点是：列表不直接承载生成设定按钮，列表只负责发现和进入详情；生成设定在小说详情页作为下一步推荐动作出现。

### 下一轮建议评审

下一轮评审链路 2：生成设定与确认设定。

重点看：

- 生成设定在哪里发起。
- 任务进度在哪里看。
- 生成结果是候选还是正式版本。
- 用户如何确认、重试、继续优化或放弃。
- 确认设定后如何进入大纲生成。

## 链路 2：生成设定与确认设定

### 业务目标

用户从已创建小说进入详情页后，一键生成小说设定档案。系统把设定生成为候选版本并自动审稿，用户看摘要、评分和风险后决定确认、优化、重新生成、放弃或强制继续。确认后，设定成为当前正式设定，小说进入“待生成全书大纲”。

### 当前页面承接

| 环节 | 当前设计 | 结论 |
| --- | --- | --- |
| 入口 | 小说列表只提供“详情”，进入小说详情后推荐“生成小说设定” | 闭环 |
| 发起任务 | 小说详情工作台顶部推荐动作或设定区触发 `novel_setting_generate` | 闭环 |
| 任务进度 | 通过详情页最近任务区、任务抽屉展示进度和失败 | 部分闭环 |
| 结果保存 | 设定结果保存为候选版本，不自动成为正式设定 | 闭环 |
| AI 审稿 | 设定生成后自动保存审稿报告，低分进入阻塞或风险确认 | 闭环 |
| 用户确认 | 设定确认抽屉或设定区展示摘要、评分、风险、下一步 | 部分闭环 |
| 采用后承接 | 当前设定版本切换为正式版本，推荐动作变为“生成全书大纲” | 闭环 |

### 已经闭环的点

- 设定生成依赖已确认方向，不会在创建页或空项目里直接生成。
- 设定是关键资产，默认候选待确认，AI 不能静默覆盖正式设定。
- 设定确认后才能生成全书大纲，前置门禁明确。
- 低分设定可以强制继续，但必须填写原因。
- 采用设定后会记录决策、切换版本、刷新推荐动作，并生成大纲可用摘要。

### 发现的缺口

| 缺口 | 影响 | 本轮处理 |
| --- | --- | --- |
| 旧文档仍写“从小说列表推荐动作进入确认抽屉” | 和列表只显示“详情”的规则冲突 | 已改为从详情页或详情页内抽屉承接 |
| 小说详情原型只列出设定操作，没有逐状态说明 | 小白用户可能不知道生成中、待确认、失败、阻塞分别点什么 | 已补充设定区状态承接表 |
| 任务进度的默认入口需要更具体 | 用户看到“生成中”时可能不知道在哪里查看进度 | 已要求详情页最近任务区和任务抽屉都能查看 |
| 设定确认抽屉的“采用后下一步”需要明确 | 用户确认后可能不知道下一步生成大纲 | 已补充确认后自动切到 `outline/not_started`，推荐动作变为“生成全书大纲” |

### 本轮结论

链路 2 为“部分闭环，已补齐主要缺口”。

业务规则已经比较完整，核心风险在页面承载表达：用户必须在详情页看到当前设定阶段的明确状态和下一步动作，不能只看到一个“生成设定”按钮。设定生成、任务进度、候选确认、低分强制确认、采用后进入大纲的链路已经具备闭环基础。

### 下一轮建议评审

下一轮评审链路 3：生成全书大纲、阶段大纲与章节目录。

重点看：

- 全书大纲、阶段大纲、章节目录是否拆成三个确认点。
- 阶段数量调整后，章节目录如何重算。
- 大纲候选如何确认、优化、放弃。
- 确认章节目录后如何进入试写。
- 已有下游内容时修改大纲如何提示影响。

## 链路 3：生成全书大纲、阶段大纲与章节目录

### 业务目标

用户确认设定后，按“全书大纲 → 阶段大纲 → 章节目录”逐层生成和确认创作骨架。每一层都是候选资产，用户确认后才成为正式版本；章节目录确认后进入试写。

### 闭环判断

| 环节 | 当前设计 | 结论 |
| --- | --- | --- |
| 入口 | 小说详情大纲页签承接 | 闭环 |
| 三层结构 | 需求文档明确三层候选确认 | 闭环 |
| 阶段数量调整 | 已有规则，但原型只显示“调整阶段数量”按钮 | 部分闭环 |
| 候选确认 | 文档明确候选确认，原型展示不足 | 部分闭环 |
| 下游影响 | 文档要求已有下游内容时提示影响，原型未展示 | 部分闭环 |
| 下一步承接 | 章节目录确认后进入试写 | 闭环 |

### 发现的缺口

| 缺口 | 影响 | 本轮处理 |
| --- | --- | --- |
| 大纲页签只展示章节目录，用户看不到全书大纲和阶段大纲是否已确认 | 小白不知道当前骨架做到哪一层 | 需要在原型补三层大纲状态卡 |
| 阶段数量调整没有说明会重算哪些内容 | 用户可能误以为只是改一个数字 | 需要弹窗展示影响范围 |
| 大纲修改和下游内容影响没有提示 | 已试写或已生成正文后改大纲会造成隐性错乱 | 需要在大纲区展示“下游影响提示” |

### 本轮结论

链路 3 为“部分闭环”。

业务规则完整，但原型承载不足。大纲页不能只像普通章节表格，需要让用户看到全书大纲、阶段大纲、章节目录三个确认点，以及调整阶段数量后的影响。

## 链路 4：试写章节与试写调试

### 业务目标

用户确认章节目录后，先生成前 1-3 章试写并让 AI 审稿。系统给出试写总评，用户可以优化试写章节、重新试写、返回上游修改，或确认进入批量正文生成。

### 闭环判断

| 环节 | 当前设计 | 结论 |
| --- | --- | --- |
| 入口 | 小说详情试写页签 | 闭环 |
| 试写生成 | 需求明确任务化生成和审稿 | 闭环 |
| 试写总评 | 原型有总评，但动作偏少 | 部分闭环 |
| 优化试写章节 | 可跳转章节详情处理 | 闭环 |
| 进入批量正文 | 原型没有明确确认入口 | 部分闭环 |
| 首条测试视频 | 需求明确是运营测试分支，不等于待视频化 | 闭环 |

### 发现的缺口

| 缺口 | 影响 | 本轮处理 |
| --- | --- | --- |
| 试写通过后没有“确认试写，进入批量正文”动作 | 用户不知道什么时候可以开始生成正文 | 需要在试写页补确认入口 |
| 低分试写只有优化按钮，缺少返回设定/大纲的路径 | 如果问题来自上游，用户只能在章节里打补丁 | 需要补“返回调整目录/大纲”入口 |
| 首条测试卡容易和正式视频化混淆 | 可能误以为试写后就能正式创建视频 | 原型需标明“测试，不改变小说待视频化状态” |

### 本轮结论

链路 4 为“部分闭环”。

试写调试主路径成立，但原型需要补“确认试写通过”和“返回上游调整”的可见动作，避免用户只看到优化章节。

## 链路 5：批量生成正文与任务进度

### 业务目标

试写确认后，用户在小说详情章节页发起批量正文生成。系统按章节目录生成正文，展示父任务和章节子任务进度；失败、取消、重试和待确认结果都能回到可操作状态。

### 闭环判断

| 环节 | 当前设计 | 结论 |
| --- | --- | --- |
| 发起位置 | 小说详情“章节”页签发起 | 闭环 |
| 列表边界 | 列表仍出现“查看生成进度”行按钮 | 部分闭环 |
| 任务抽屉 | 有任务表格，但缺少失败处理和子任务摘要 | 部分闭环 |
| 章节结果 | 需求要求候选或正式版本有确认规则 | 闭环 |
| 中途失败 | 任务文档完整，原型展示不足 | 部分闭环 |
| 下一步承接 | 完成后进入全书审稿 | 闭环 |

### 发现的缺口

| 缺口 | 影响 | 本轮处理 |
| --- | --- | --- |
| 小说列表行按钮展示“查看生成进度” | 列表承接了详情页工作台能力 | 需要列表只保留“详情”和“查看任务” |
| 任务抽屉没有显示失败数、待确认数、当前章节和重试动作 | 批量任务异常时用户不知道怎么处理 | 需要补任务摘要和失败操作 |
| 批量生成在列表工具栏仍有按钮 | 用户绕过试写确认和章节页上下文 | 需要从列表工具栏移除 |

### 本轮结论

链路 5 为“部分闭环”。

正确的业务入口在小说详情章节页；列表只能做筛选、进入详情和查看任务摘要。原型要把批量正文生成从列表收回详情页，并强化任务抽屉。

## 链路 6：单章正文查看、编辑、重写和候选采用

### 业务目标

用户进入章节详情后，可以查看当前正式正文、章节摘要、AI 建议、历史版本；可以手动编辑正文，也可以让 AI 生成新改稿候选。候选需先保存，采用前进行影响评估。

### 闭环判断

| 环节 | 当前设计 | 结论 |
| --- | --- | --- |
| 章节导航 | 左侧章节导航，右侧内容区 | 闭环 |
| 正文查看 | 当前正文、版本、字数可见 | 闭环 |
| 正文编辑 | 已有编辑、保存、取消 | 闭环 |
| 历史版本 | 版本标签可打开版本抽屉，可回退生成新版本 | 闭环 |
| AI 新改稿 | 弹窗展示可编辑候选字段和正文 | 闭环 |
| 候选采用 | 有采用入口，但影响决策还不完整 | 部分闭环 |

### 发现的缺口

| 缺口 | 影响 | 本轮处理 |
| --- | --- | --- |
| “保存候选”含义容易误解 | 用户可能以为保存候选就是采用正式正文 | 原型应保持候选卡片展示，并在文案里说明“保存候选不是采用” |
| 采用候选后的影响处理只有单一确认 | 不足以表达轻微/中等/严重影响策略 | 交给链路 7 补齐 |

### 本轮结论

链路 6 为“基本闭环”。

章节详情工作台已经覆盖查看、编辑、版本、生成候选、候选展示。剩余关键在采用后的影响处理选择。

## 链路 7：章节重写影响后文处理

### 业务目标

用户采用章节改稿或回退历史版本后，AI 评估是否影响后续章节。轻微影响只记录备注；中等影响标记受影响章节，阻塞完成；严重影响提供清空后续、标记逐章处理或返回修改当前候选的选择。

### 闭环判断

| 环节 | 当前设计 | 结论 |
| --- | --- | --- |
| 影响触发 | 采用候选和回退版本会触发 | 闭环 |
| 影响等级 | 文档完整，原型只展示中等影响 | 部分闭环 |
| 用户选择 | 原型只有“确认并标记后续章节” | 部分闭环 |
| 受影响章节 | 小说详情章节页缺少影响摘要 | 部分闭环 |
| 完成门禁 | 文档明确未处理影响会阻塞完成 | 闭环 |

### 发现的缺口

| 缺口 | 影响 | 本轮处理 |
| --- | --- | --- |
| 影响弹窗没有三种处理选择 | 用户无法决定是同步、标记逐章处理还是清空后续 | 需要在章节详情补三选项 |
| 小说详情里看不到影响案例摘要 | 用户回到小说详情后不清楚哪些章节被影响 | 需要在章节页签补影响处理摘要 |

### 本轮结论

链路 7 为“部分闭环”。

规则已经清楚，但原型必须让用户完成决策，而不是只看到一个确认按钮。

## 链路 8：全书审稿、优化和完成确认

### 业务目标

正文全部完成并且没有未处理章节后，用户发起全书 AI 审稿。系统输出全书评分、问题、风险和完成门禁；通过后用户确认小说完成，低分继续必须记录原因。

### 闭环判断

| 环节 | 当前设计 | 结论 |
| --- | --- | --- |
| 发起全书审稿 | 全书审稿页签存在 | 闭环 |
| 门禁条件 | 文档完整 | 闭环 |
| 审稿报告 | 原型只有简单描述 | 部分闭环 |
| 问题处理 | 文档完整，原型缺动作 | 部分闭环 |
| 确认完成 | 原型缺确认完成弹窗 | 部分闭环 |
| 低分强制继续 | 文档完整，原型缺原因入口 | 部分闭环 |

### 发现的缺口

| 缺口 | 影响 | 本轮处理 |
| --- | --- | --- |
| 全书审稿页签不能完成“审稿 → 处理问题 → 确认完成”闭环 | 用户不知道怎么从审稿走到完成 | 需要补门禁卡、问题列表和确认完成按钮 |
| 强制继续没有原因记录入口 | 不符合高风险操作记录要求 | 需要补接受风险弹窗 |

### 本轮结论

链路 8 为“部分闭环”。

需求文档已经完整，原型需要补完成门禁和确认完成动作。

## 链路 9：待视频化确认与视频引用状态

### 业务目标

用户确认小说完成后，系统进行待视频化检查。检查通过后，小说进入“可被视频引用”。创建正式视频项目必须在视频模块发起，小说模块只展示状态、快照和跳转入口。

### 闭环判断

| 环节 | 当前设计 | 结论 |
| --- | --- | --- |
| 视频化检查 | 文档完整 | 闭环 |
| 小说列表展示 | 原型已有视频引用状态列，但 mock 文案仍是“可创建视频” | 部分闭环 |
| 小说详情承接 | 视频引用页签存在，但内容太轻 | 部分闭环 |
| 视频模块创建 | 视频列表有“新建视频”弹窗 | 闭环 |
| 引用异常 | 文档完整，原型已有标签但缺快照说明 | 部分闭环 |

### 发现的缺口

| 缺口 | 影响 | 本轮处理 |
| --- | --- | --- |
| 小说列表还出现“创建视频项目”动作 | 混淆小说系统和视频系统边界 | 需要移除小说列表创建视频按钮 |
| 视频引用状态文案应是“可被引用/已引用/引用异常” | “可创建视频”听起来像小说列表能创建视频 | 需要统一文案 |
| 小说详情视频区需要展示视频化快照和去视频列表入口 | 用户不知道视频项目在哪里创建 | 需要补快照和跳转 |

### 本轮结论

链路 9 为“部分闭环”。

业务边界已经确定：小说模块只管理可引用状态，视频模块负责创建视频。

## 链路 10：AI 任务失败、待确认和重试

### 业务目标

所有 AI 任务都能展示正在做什么、进度、失败原因、待确认结果和下一步动作。失败不会让小说卡死；重试、换模型、取消和放弃结果都有记录。

### 闭环判断

| 环节 | 当前设计 | 结论 |
| --- | --- | --- |
| 任务状态 | 文档统一 6 个主状态 | 闭环 |
| 任务入口 | 详情页任务抽屉、任务中心 | 闭环 |
| 列表摘要 | 列表可以查看任务摘要 | 闭环 |
| 失败处理 | 文档完整，原型任务抽屉偏弱 | 部分闭环 |
| 待确认结果 | 文档完整，各业务页局部承接 | 部分闭环 |
| 任务详情 | 任务中心已有基础表格 | 部分闭环 |

### 发现的缺口

| 缺口 | 影响 | 本轮处理 |
| --- | --- | --- |
| 任务抽屉缺少失败重试、换模型重试和取消入口 | 失败后用户只能关闭抽屉 | 需要补任务抽屉操作 |
| 批量任务缺少成功/失败/待确认数量 | 用户看不懂批量正文生成实际进展 | 需要补任务摘要 |

### 本轮结论

链路 10 为“部分闭环”。

任务规则足够完整，原型需要把失败动作和批量摘要表达出来。

## 链路 11：暂停、归档、恢复和复盘

### 业务目标

用户可以暂停或归档小说，必须记录原因；恢复时回到原创作阶段并重新计算推荐动作；归档不删除资产，不影响已创建视频引用快照。

### 闭环判断

| 环节 | 当前设计 | 结论 |
| --- | --- | --- |
| 暂停 | 文档有规则，原型未显性展示 | 部分闭环 |
| 归档 | 列表曾有归档弹窗，但入口应进入详情内 | 部分闭环 |
| 恢复 | 文档有规则，原型未展示 | 部分闭环 |
| 原因记录 | 文档明确必须填写 | 闭环 |
| 视频引用保护 | 文档完整 | 闭环 |

### 发现的缺口

| 缺口 | 影响 | 本轮处理 |
| --- | --- | --- |
| 归档入口在列表“更多”容易误触 | 归档是高风险动作，应放详情内 | 需求口径保持：列表不放归档主动作 |
| 暂停/恢复不是 P0 主流程 | 原型无需本轮大做，但需要文档保留闭环 | 本轮只在评审中记录，不强行扩展页面 |

### 本轮结论

链路 11 为“部分闭环，非本轮原型重点”。

归档、暂停、恢复属于生命周期辅助能力。文档规则清楚，原型可以后续在详情页“更多操作”补齐。

## 本轮总体结论

| 链路 | 结论 | 是否本轮调整原型 |
| --- | --- | --- |
| 1 创建小说到待生成设定 | 部分闭环，已补主要文档口径 | 否，前轮已调整 |
| 2 生成设定与确认设定 | 部分闭环，已补主要文档口径 | 是，详情页需更清晰 |
| 3 大纲与目录 | 部分闭环 | 是 |
| 4 试写调试 | 部分闭环 | 是 |
| 5 批量正文与任务进度 | 部分闭环 | 是 |
| 6 单章正文与候选采用 | 基本闭环 | 小调整 |
| 7 重写影响处理 | 部分闭环 | 是 |
| 8 全书审稿与完成确认 | 部分闭环 | 是 |
| 9 待视频化与视频引用 | 部分闭环 | 是 |
| 10 任务失败与重试 | 部分闭环 | 是 |
| 11 暂停归档恢复 | 部分闭环，非本轮重点 | 暂不调整 |

本轮需要调整的原型点：

- 小说列表：只保留“详情”和“查看任务”类轻入口；移除列表中的“批量生成正文”和“创建视频项目”承载。
- 小说列表：视频列只展示视频引用状态，不表达创建视频能力。
- 小说详情：设定、大纲、试写、章节、全书审稿、视频引用都要能看到当前状态和下一步动作。
- 小说详情：批量正文生成、任务进度、失败重试应在详情页承接。
- 章节详情：影响评估弹窗补三种处理选择，候选保存和采用边界保持清楚。

## 本轮原型调整结果

已按上述评审结果调整第一版原型：

| 调整项 | 结果 |
| --- | --- |
| 小说列表边界 | 行主按钮收敛为“详情”，列表不再展示创建视频项目或查看生成进度类主动作 |
| 表格操作区 | 保留创建小说、批量审稿；正文批量生成回到小说详情章节管理区 |
| 视频引用状态 | mock 和列表筛选统一为“未准备/可被引用/已引用/引用异常” |
| 小说详情大纲区 | 补充全书大纲、阶段大纲、章节目录三层状态卡和阶段数量调整弹窗 |
| 小说详情试写区 | 补充优化试写、返回目录、生成首条测试卡、确认试写进入批量正文 |
| 小说详情章节区 | 补充批量任务摘要和影响处理摘要 |
| 小说详情全书审稿区 | 补充问题卡、重新审稿、接受风险原因和确认小说完成弹窗 |
| 小说详情视频引用区 | 补充视频化检查、可引用范围、引用快照和跳转视频列表 |
| 任务抽屉 | 补充当前对象、成功/失败/待确认数量、重试、换模型重试、取消 |
| 章节详情影响处理 | 补充轻量调整、标记受影响章节、清空后续正文重来三种选择 |

本轮没有展开暂停、归档、恢复页面原型；该链路已进入需求文档，后续可在小说详情“更多操作”中补齐。
