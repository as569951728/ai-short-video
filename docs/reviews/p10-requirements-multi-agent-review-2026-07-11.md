# P10 下一阶段需求多 Agent 评审

## 评审方式

主控分别从四个独立视角评审 P10，评审阶段不修改业务代码、不派发实现：

- 产品流程与普通用户体验
- 技术架构与可落地性
- 测试验收与边界证明
- 工程质量、安全与审计

## 一致结论

1. P10 是人工发布事实记录和手动数据回填，不是自动发布。
2. P9e 导出记录不能复用为发布记录；必须有独立 PublishRecord 和不可覆盖的冻结快照。
3. 发布后同步创建 24h/48h 待回填节点，逾期通过读侧时间计算，不使用定时器。
4. 多平台和多次发布必须是独立记录；业务重复和幂等重复需要区分。
5. 数据更正、撤销、样本不足和下一步决策必须可审计，不能静默覆盖。
6. P10 不接平台 token、Cookie、账号授权、平台 API、自动上传、AI 分镜或 P12 看板。
7. 未授权真实 MySQL smoke 时，只能声明 mock/local、in-memory、schema/static 和 API contract 通过。
8. 共享工作树在 P10 开工前必须刷新归因检查点。

## 分歧与主控决策

### 创作来源合同是否阻塞 P10

- 产品和架构视角：不阻塞最小发布记录，但应独立修复。
- 测试视角：建议作为 P10-preflight P1，因为后续来源归因会失真。
- 安全视角：若 P10 只记录发布事实可不阻塞；若宣称来源维度运营回流则必须先修。

主控决策：独立作为 `P10-preflight` 先关闭，不混入 P10 主包。理由是页面已经向用户承诺三种创作来源，但当前请求合同完全丢失该选择；先修复能避免后续数据含义不一致，也不会污染 P10 发布模型。

### 24h/48h 是否需要后台任务

主控决策：不使用定时器。发布记录创建时同步建立两个节点，查询时根据 `publishedAt + 24h/48h` 惰性计算 waiting/due/overdue。

### 平台、账号和链接

主控决策：平台使用固定枚举加 `other`；账号使用自由文本展示名；作品链接或作品 ID 至少填写一个。P10 不建立平台账号中心。

### 表现好坏阈值

主控决策：P10 不自动判定爆款或失败。仅记录样本是否充足、主观评价和用户选择的下一步动作。

## 需求准确性门槛

每个后续阶段继续采用至少三个独立评审视角：产品流程、技术架构、测试验收；涉及数据、权限、外部平台或真实环境时必须增加安全/工程质量视角。主控负责解决分歧并将决策写入正式需求，评审结论不能只保留在会话中。

## 成稿复核修订

正式成稿经产品、架构、测试三视角复核后，主控采纳以下修订：

- P10 不把任务包 11 才引入的封面文案作为现阶段硬门禁。
- PublishRecord 和冻结快照显式保存并校验安全的 `sourceVersionRefs`。
- 增加可执行 fixture 清单、显式阻塞标准和删除/归档保护验收入口。
- 将“已复盘”改为“已记录下一步决策”，避免越界暗示 P12 复盘看板。

修订完成后，产品流程、技术架构和测试验收三视角均给出 `confirmed`。P10 当前状态为：需求已确认，等待主控获得用户授权后先执行 `P10-preflight`，尚未派发 P10 实现。
