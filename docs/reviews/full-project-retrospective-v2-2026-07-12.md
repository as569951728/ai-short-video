## 1. 二次复盘目标

第一次复盘完成了问题发现、能力边界纠偏和路线判断，但仍存在重复 ID、状态覆盖、责任过宽、任务不可直接派发和验收证据不均匀等问题。

二次复盘只做五件事：

1. 将所有发现归并成唯一问题总账。
2. 将确认缺陷、验证缺口和工程债务分开。
3. 为每项问题绑定责任、依赖、整改包和验收用例。
4. 建立关闭证据模板，禁止用“代码已写”代替“问题已关闭”。
5. 冻结新需求和新业务研发，直到整改总账全部关闭。

## 3. 唯一状态与分类

### 3.1 状态

| 状态 | 定义 |
| --- | --- |
| open | 已确认问题，尚未实现 |
| implemented_pending_verification | 已有实现，但没有达到对应证据等级 |
| partial | 部分场景关闭，仍有明确未证明项 |
| verification_gap | 尚未完成专项审计，不能判断为缺陷或通过 |
| closed | 原始场景、回归场景和对应证据等级均通过 |

任何问题只有 `docs/remediation/issue-ledger.md` 可以更新正式状态。其他评审文档保留历史，不再通过追加段落覆盖状态。

### 3.2 问题类别

| 类别 | 含义 | 开启新需求前要求 |
| --- | --- | --- |
| PB | Product Blocker，用户核心目标无法完成或结果不可信 | 必须关闭 |
| RB | Release Blocker，真实环境、并发、持久化或媒体能力不成立 | 必须关闭 |
| QG | Quality Gate，验收、防复发、安全或运维证据缺失 | 必须关闭 |
| DEBT | Engineering Debt，当前未直接阻塞但会扩大回归和维护成本 | 必须按总账关闭，不接受无限期延期 |

严重度 P0/P1/P2 只表示修复顺序，不再替代问题类别。

冲突状态采用最保守证据原则：

- 代码存在、浏览器未复验：`implemented_pending_verification`。

- 原始故障通过但系统性机制未完成：个案可记录通过，正式问题保持 `partial`。
- 没有专项审计：`verification_gap`，不得写成“无问题”。

## 5. 二次复盘新增遗漏范围

第一次复盘未充分覆盖以下领域。它们先进入验证缺口，不预判一定存在缺陷：

| ID | 领域 | 必须回答的问题 |
| --- | --- | --- |
| RMD-AUD-SEC-001 | 权限与租户 | route/service/repository 是否都有 tenant filter；是否存在越权读取或跨租户引用 |
| RMD-AUD-OPS-001 | 部署与回滚 | 环境配置、migration 顺序、health、回滚和故障恢复是否可执行 |
| RMD-AUD-OBS-001 | 观测与 SLO | 长任务是否有 latency、成功率、失败分类、卡死告警和关联 taskId |
| RMD-AUD-COST-001 | 成本与容量 | 模型 tokenUsage、并发、TTS、渲染、存储和下载成本是否有预算与限流 |
| RMD-AUD-DR-001 | 数据备份与恢复 | 创作资产、版本、任务、发布记录是否可备份、恢复和校验 |
| RMD-AUD-A11Y-001 | 可访问性与响应式 | 键盘、焦点、屏幕阅读器、窄屏和长文本是否可用 |
| RMD-AUD-SUPPLY-001 | 依赖与供应链 | 依赖漏洞、许可证、锁文件、运行时版本和构建可复现性是否受控 |

这些问题不要求在整改前立即建设完整平台；必须先完成审计，再按证据关闭或拆出确认缺陷。

## 6. 根因二次归并

### RC2-01 价值流未成为完成定义

技术包按 DTO、API、页面和状态机关闭，但没有以“完成一本真实小说”和“得到一个真实视频文件”为最终完成定义。

### RC2-02 真实环境门禁后置

mock、in-memory 和 static 先行是合理策略；问题在于没有提前冻结从 E4 升到 E5/E6/E7 的必要条件和排期。

### RC2-03 AI 动作缺平台级生命周期

页面分别补 loading、localStorage、轮询和重试，服务端却没有统一 worker、checkpoint、取消、重试和迟到结果处理。

### RC2-04 验收围绕实现层而非可观察结果

按钮存在、接口成功、数据非空被当作通过；没有稳定断言结果位置、下一动作、刷新、多标签页和真实产物。

### RC2-05 状态、文档和 Git 都缺单一事实源

同一问题在多份报告中有不同状态；多包改动长期未提交；旧巡检口径可覆盖当前代码事实。

## 7. 二次复盘主控结论

1. 第一次复盘的方向性结论成立，但执行状态以唯一问题总账为准。
2. 当前不允许进入新的需求设计、P10-R1 或其他业务研发。
3. 整改不是一次大重构；RP-00 至 RP-10 只管理依赖，实际执行必须使用可派发子包，逐包提交、测试和关闭。
4. 小说真实完本仍是首要价值目标，视频真实媒体在小说金丝雀后执行。
5. 所有确认问题与验证缺口全部关闭后，主控才可重新进入下一阶段需求设计。

## 8. 完成定义

二次复盘落地完成需同时满足：

- 唯一问题总账没有 `open`、`partial`、`implemented_pending_verification` 或 `verification_gap`。
- 每个问题至少有一个 acceptance id 和独立测试结论。
- PB/RB 达到目标环境证据等级，不能用 mock 代替。
- QG 已形成可重复执行的脚本、用例或流程门禁。
- DEBT 已完成定向拆分或治理，不以“以后再做”关闭。
- 每个整改包有独立 commit、远程 push 和主控收口记录。
