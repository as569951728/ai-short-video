# P10-preflight 创作来源多 Agent 评审

## 评审方式

主控在不改业务代码、不派实现的前提下，分别从以下视角独立评审：

- 产品流程与普通用户体验。
- 技术架构、接口、数据和迁移可落地性。
- 测试 fixture、边界和阻塞门禁。

## 一致结论

1. 创作来源不能只保留在前端选中态，必须进入请求、领域校验、持久化、列表/详情和刷新回显。
2. 系统推荐允许可选补充想法；手动想法必须填写核心想法；两者不能携带热点引用字段。
3. 引用热点必须选择真实、同租户、可访问的报告或机会点；没有可选数据时必须禁用，不能做假交互。
4. 旧数据不能默认显示为用户选择过系统推荐，应使用 `legacy_unknown` / “未记录来源”。
5. mock、内存仓储和静态 migration 通过不能替代真实 MySQL/Prisma live smoke。
6. 本前置项不建设热点管理、来源运营分析、P10 发布记录或 P12 看板。

## 分歧与主控决策

### 热点服务不可用时是否允许保存未验证引用

- 架构视角提出可保存 `unverified_service_unavailable`，以兼容热点服务暂不可用。
- 产品和测试视角认为这会把无法验证的 ID 伪装成真实引用。

主控决策：不允许。没有真实可选热点或无法验证时，前端禁用并由后端拒绝 `hotspot_reference`。后续若建设正式热点服务，再扩展可用性策略。

### 历史数据是否默认系统推荐

- 为兼容旧客户端，新创建请求缺少字段时可在过渡期默认 `system_recommendation`。
- 历史数据库行缺少来源时不能据此推断用户选择。

主控决策：请求兼容默认和历史事实分开处理。历史行使用 `LEGACY_UNKNOWN`；界面显示“未记录来源”。

### 来源字段保存位置

主控决策：以 `CreateNovelPreferences.creationSourceType` 为权威来源，不在 `Novel` 重复双写。列表通过关联或安全投影返回来源摘要，后续只有明确性能证据时才考虑冗余索引字段。

### 手动想法最小长度

主控决策：去除首尾空格后不少于 6 个字符；错误落到核心想法字段。系统推荐和热点引用中的补充想法保持可选。

## 成稿状态

正式合同：`docs/modules/novel-creation-source-contract.md`。

首轮正式成稿终审结果：产品和测试 `confirmed`；架构要求消除“新前端必填 / 旧客户端兼容缺省”的表述冲突，并明确数据库 `LEGACY_UNKNOWN` 到 API `legacy_unknown` 的映射。主控完成修订后，架构复核为 `confirmed`。

最终结论：产品、架构、测试三视角全部 `confirmed`。当前状态为“需求已确认，等待用户授权研发”；未派发实现。

## 开工包补充复核

主控核对仓库后确认当前没有热点查询模块，热点分析菜单仍禁用。为避免假数据和假交互，正式合同与研发包补充了 `HotspotReferenceGateway`：默认运行实现返回不可用，页面禁用引用热点，API 拒绝；测试通过注入 fake gateway 覆盖合法、缺失、跨租户和归属错误。

该补充由产品、架构、测试再次只读复核，三方均为 `confirmed`。正式派发包见 `docs/modules/novel-creation-source-implementation-package.md`。
